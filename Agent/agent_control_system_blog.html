1为什么需要控制系统视角？

过去我们常把 Agent 系统描述成 LLM + Tools + Memory。这个描述能解释 Agent 的组成，却不足以解释 Agent 如何稳定工作。真正进入生产场景后，关键问题不是“它能不能生成答案”，而是“它能不能在复杂、动态、带噪声的环境中持续逼近目标，并且不失控”。

在前面的讨论里，我们曾经用操作系统类比 Agent：Agent 像进程，工具调用像系统调用，Agent Runtime 像内核，权限、隔离、调度、审计都是基础设施。这个视角非常适合回答一类问题：

  • Agent 能调用哪些工具？
  • Agent 的权限边界在哪里？
  • 多个 Agent 如何隔离和调度？
  • 如何记录、审计、回放一次执行？

但它还没有完整回答另一个问题：Agent 如何稳定地把系统从当前状态推进到目标状态？

这正是控制系统关心的问题。一个控制系统不会只看“控制器聪不聪明”，而会同时看目标、状态估计、执行器、传感器、误差、反馈、扰动、稳定性和约束。把这套语言迁移到 Agent 系统后，我们会得到一个更工程化的结论：

Agent 不是一个会调用工具的聊天机器人，而是一个在不确定环境中，通过反馈持续控制系统状态、逼近目标，同时保持稳定和安全的闭环控制器。
图 1：操作系统视角与控制系统视角的分工
OS 视角更偏“运行边界”，控制系统视角更偏“目标收敛”。前者让 Agent 不越界，后者让 Agent 不发散。

2从开环生成到闭环控制

最朴素的大模型应用是开环系统：

用户输入 → LLM 生成结果 → 结束

这个结构没有持续观察，没有误差修正，也没有稳定性机制。模型一次性输出，对了就对了，错了就错了。很多早期“AI 写代码”或“AI 生成文档”的体验，问题正出在这里：它只有生成能力，没有控制能力。

真正的 Agent 应该是闭环系统：

目标 / Spec → Agent 决策 → 工具执行 → 环境变化 → 观察结果 → 误差评估 → 修正计划 → 继续执行

闭环的含义不是“多调用几次模型”，而是每一次行动之后，系统都能观测真实结果，把结果和目标比较，计算差距，再决定下一步动作。

图 2：Agent 作为闭环控制器
控制系统把“生成结果”变成“根据误差不断修正动作”。测试、监控、日志和人工 Review 都是 Agent 的传感器。

下面这张映射表可以作为全文的骨架。它把控制系统中的关键概念翻译到 Agent 系统中。

从这张表开始，Agent 系统的核心问题不再是“模型聪不聪明”，而是：

这个闭环系统能否稳定、可控、可验证地把世界从当前状态推向目标状态？

4Reference：Spec 不是文档，而是目标信号

控制器首先需要一个目标值。例如恒温器需要知道目标温度，自动驾驶需要知道目标路线、速度、安全距离。Agent 系统同样需要明确目标，而这个目标就是 Spec。

问题在于，大多数需求文档并不是好的控制目标。它们通常写给人看，依赖人的背景知识、隐性约束和工程经验。例如：

实现多 workspace 并行，每个 workspace 独立维护 Quest session，互不干扰。

这句话作为产品意图是可以的，但作为 Agent 的目标信号远远不够。Agent 还需要知道路径格式如何兼容、存量锁表任务如何处理、Worktree 场景是否支持、旧接口是否不能动、哪些行为属于非目标、如何证明完成。

因此，AI-native 开发真正需要的是 Executable Spec，或者更强一点叫 Ultra Spec：它不是简单描述功能，而是把模糊意图转换成可控制、可观测、可验证的目标信号。

图 3：从模糊意图到可执行目标信号
Ultra Spec 的价值在于把“想要什么”翻译成 Agent 可以执行、系统可以验证、人类可以审查的目标结构。
设计原则AI-native 不是 Code-first，而是 Spec-first；更准确地说，是 Executable-Spec-first。人类最重要的工作不再是亲手写每一行代码，而是在 Agent 行动前把目标、边界和验收标准定义清楚。

5State Estimation：Agent 看到的是“状态估计”，不是世界本身

控制系统中，很多真实状态无法直接测量，只能通过传感器估计。例如自动驾驶并不知道“世界真相”，只能通过摄像头、雷达、地图、GPS 融合出一个当前状态。

Agent 也一样。它看到的不是代码库、产品和业务系统的全部真相，而是一个由多种信息拼出来的 belief state：

  • 当前上下文窗口；
  • 检索到的文档和代码片段；
  • Repo Wiki、代码知识图谱和架构文档；
  • 测试结果、日志、监控指标；
  • 历史 Spec、决策记录、团队记忆；
  • 工具调用返回结果。

这个状态估计可能出错。文档可能过期，知识图谱可能没更新，检索可能漏掉关键文件，历史记忆可能已经不适用，测试也可能是 flaky 的。因此，成熟 Agent 系统不应该简单地“把更多内容塞进上下文”，而应该建立一个 Context Estimator / Belief Manager。

它至少需要做几件事：融合多源信息，标注来源，标注时间，识别冲突，判断置信度，发现过期信息，并在不确定时主动补充观测。

图 4：Agent 的状态估计器
Repo Wiki、代码知识图谱和历史决策不只是文档，它们是提高 Agent 可观测性的传感器和状态模型。

6Error 与 Feedback：测试、Review 和监控都是传感器

控制系统的核心不是动作，而是误差：

Error = Target State − Current State

在 Agent 系统中，误差信号可以来自许多地方：测试失败、类型检查失败、Lint 失败、E2E 失败、性能指标不达标、安全扫描失败、Review Agent 发现风险、用户验收不通过、线上监控异常，或者 Spec checklist 没有完成。

如果没有误差信号，Agent 就不知道自己做得对不对。很多 AI 代码生成 demo 看起来很酷，但不能进入生产，就是因为它们有生成能力，却没有误差反馈。

更进一步，一个成熟 Agent 系统应该有多层反馈回路，而不是一个单一反馈点。

反馈层级反馈来源时间尺度主要作用
工具调用级命令输出、API 返回、文件修改结果秒级确认动作是否成功执行
代码级单测、类型检查、Lint秒到分钟快速发现局部错误
PR 级集成测试、Review、静态分析分钟到小时判断能否合入
发布级监控、告警、用户反馈小时到天判断线上状态是否稳定
组织级复盘、规则沉淀、评估集更新天到周让下一次执行更好
图 5：多速率反馈回路
反馈越慢，Agent 的行动步长越要小；反馈越快，Agent 可以拥有更高自主性。

7Controller：Agent Planner 更像 MPC，而不是一次性计划器

很多 Agent 系统会让模型先生成一个完整计划，然后按计划执行。但在复杂环境中，一次性计划很容易失效：代码状态会变化，测试会暴露新问题，外部依赖会失败，需求也可能被澄清。

更适合 Agent 的控制方式是 MPC，Model Predictive Control，模型预测控制。它的基本思想是：

基于当前状态预测未来几步
生成一个短期计划
只执行第一步或少数几步
观察结果
重新估计状态
重新规划

这正是 Agent 应该采用的模式：Plan → Act → Observe → Replan。它比“一口气大改所有文件”稳定得多。

图 6：MPC 式 Agent Planner
MPC 思路让 Agent 更保守、更可控：预测多步，但只执行少数步骤，然后根据反馈重新规划。

8Gain：Agent 的自主性需要动态调参

控制系统里，增益太高会震荡，增益太低会迟钝。Agent 也有“增益”：它决定 Agent 多大胆、多主动、多大范围地修改系统。

高增益 Agent 的表现是快速行动、大胆修改、少问问题、主动重构；低增益 Agent 则小步修改、频繁确认、严格遵守 Spec、更多上抛。

问题不在于高增益或低增益谁更好，而在于不同任务应该使用不同增益。

任务类型推荐 Agent Gain原因
文档生成、注释补齐高风险低、反馈快、可回滚
单元测试补齐中高目标清晰，测试本身也是反馈
小 bug 修复、UI 微调中可自动验证，但仍需控制 diff 范围
跨模块重构中低影响面大，容易局部最优全局变差
权限系统、数据迁移、安全逻辑低不可逆风险高，需要强审批
删除历史兼容逻辑极低隐藏依赖多，必须人类主导

成熟 Agent Runtime 应该支持模式切换，而不是所有任务都一个自主性档位：

explore mode   ：大胆调研，少写入
implement mode ：受控实现，小步提交
review mode    ：保守检查，独立上下文
repair mode    ：局部修复，限制 diff
incident mode  ：只读优先，谨慎操作
图 7：Agent 自主性应随风险和反馈质量动态调整
自主性不是固定开关，而是根据风险、反馈质量、目标清晰度动态调度的控制参数。

9Stability：Agent 最大的问题不是出错，而是不收敛

控制系统最怕不稳定。恒温器如果调得太激进，就会“太冷 → 猛烈加热 → 太热 → 猛烈制冷 → 太冷”，形成震荡。Agent 也会震荡。

常见表现包括：

  • 修一个测试，坏三个测试；
  • 改了 A，又改回 A；
  • 两个 Agent 互相覆盖对方代码；
  • Review Agent 和 Coding Agent 反复争执；
  • Agent 不断重构，却没有让任务更接近完成；
  • Auto-Heal 越修越大，最后超出原始需求范围。

因此，Agent Runtime 必须内置阻尼机制：

  • 限制最大步数、最大工具调用次数和最大 diff 范围；
  • 采用小步提交、checkpoint 和 rollback；
  • 对高风险文件启用只读或审批模式；
  • 连续失败后降低自主性，停止自动修复并上抛人类；
  • 用单调进展指标判断“是否真的更接近目标”。

一个非常有用的类比是 Lyapunov function：为任务定义一个“未完成度”指标，要求 Agent 的每一步动作尽量让它下降。

V = 未满足验收项数量
  + 失败测试数量
  + 高风险 Review 问题数量
  + 未解决开放问题数量
  + 未通过安全检查数量
  + Diff 风险分

如果 V 连续上升，或者 V 没有下降但代码改动越来越大，系统就应该停止、回滚或上抛。

稳定性原则每一步不仅要问“Agent 做了什么”，还要问“系统是否更接近目标”。忙碌不等于收敛，自动修复不等于稳定。

10Constraints 与 Safety Filter：受约束的智能才可生产化

控制系统不能为了更快到达目标而违反安全边界。自动驾驶不能闯红灯，工业控制不能让压力超过安全阈值。Agent 同样不能只优化“完成任务”。

它必须在约束下行动：

maximize: task_success
subject to:
  security_policy
  permission_scope
  budget_limit
  compatibility_constraints
  human_approval_required_for_irreversible_actions
  rollback_available

这意味着所有高风险动作前都需要 Safety Filter：权限检查、参数校验、风险分类、模拟执行、人工审批、审计记录和回滚计划确认。

11多 Agent 是分布式控制系统，不是群聊

一旦进入多 Agent，我们面对的就不是单控制器问题，而是分布式控制问题。前端 Agent、后端 Agent、测试 Agent、Review Agent、文档 Agent 都在影响同一个系统状态。

这会带来一系列新问题：

  • 多个 Agent 是否共享同一个目标？
  • 它们的状态估计是否一致？
  • 是否会同时修改同一文件或同一接口？
  • 局部最优是否会导致全局复杂度上升？
  • 冲突发生时由谁仲裁？

因此，多 Agent 不能只是“几个角色在一个聊天室里对话”。工程化的多 Agent 应该是：

Spec → Task DAG → Agent Assignment → Isolated Context → Artifact Handoff → Verification

全局控制器负责拆任务、识别依赖、控制并行边界、处理冲突和上抛不可逆决策；局部 Agent 在各自上下文中完成前端、后端、测试、调研、审查等专项任务。

图 9：多 Agent 作为分布式控制系统
多 Agent 的关键不是“更多角色”，而是全局控制器、任务 DAG、上下文隔离、结构化交接和全局验证。

12真实实践的控制系统解释：从认知地基到 Auto-Heal

前面讨论过的真实 AI 开发实践，给了一个非常好的校准样本。这个案例里，团队没有直接让几十个 Agent 上来写代码，而是先刷新 Repo Wiki、补全代码知识图谱、整理历史 Spec 和决策原文。用控制系统语言说，这是在建立 Plant Model 和 State Estimator。

随后他们通过 Ultra Spec 把目标、边界、兼容性、风险和验证方式明确下来。这是 Reference Signal。再通过 Team Lead Agent 按 DAG 拆任务，分配给前端、后端、测试、审查、调研等专家 Agent。这是分布式控制。最后通过 Ultra Review 和 Nightly Auto-Heal，把代码生成能力接入验证与自愈闭环。

实践中的组件控制系统含义
Repo Wiki被控对象的结构模型
代码知识图谱依赖、接口、数据流的状态模型
历史 Spec / 决策原文系统历史状态与隐性约束
Ultra SpecReference signal + constraints
Team Lead Agent高层控制器 / 调度器
专家 Agent分布式局部控制器
测试 / Review AgentSensor + verifier
Ultra Review合入前稳定性检查
Nightly Auto-Heal自动纠偏闭环
Expert Skill自适应控制参数和经验沉淀
人类拍板Supervisory control

这个案例真正值得关注的不是“生成了多少行代码”，而是它显示出一套 Agentic Engineering Operating Model：认知先行，规格驱动，分布式执行，验证闭环，自愈修复，经验沉淀。

图 10：AI-native 开发的闭环控制流程
AI-native 开发不再是“需求 → 编码 → 测试 → 上线”的直线，而是围绕状态、误差和反馈持续收敛的闭环。

13从控制系统看 Agent 的失败模式

控制系统视角最有价值的地方之一，是它能帮我们分类 Agent 失败模式，并给出对应的系统性解法。

失败模式表现系统性解法
开环幻觉一次性生成看似合理但实际错误的结果强制验证闭环
传感器缺失没有测试、日志或监控，不知道自己错了补观测能力
状态估计错误基于过期文档、错误记忆或缺失上下文行动来源、时间戳、置信度、冲突检测
控制增益过高过度重构、越界修改、修复震荡小步提交、diff 限制、审批 gate
积分饱和历史上下文太重，过度响应长期问题记忆裁剪、过期、作用域限制
正反馈失控错误经验被写入记忆并影响后续 Agent知识入库前验证、可撤销、可版本化
多控制器冲突多个 Agent 同时改同一模块，互相覆盖DAG、文件锁、接口契约、中心仲裁
反馈延迟完整 CI 太慢，Agent 在错误方向上走太远快慢反馈分层，小步推进
执行器饱和工具权限不足、API 限流、预算耗尽任务开始前评估可控性
目标函数错误完成表面指标，却违背真实意图Non-goals、SLO、验收测试、人类监督

14面向控制的 Agent Runtime 参考架构

如果完全按照控制系统思路重新设计 Agent Runtime，它不应该只是 LLM、Tools 和 Memory 的松散组合，而应该包含 Reference Manager、State Estimator、Controller、Safety Filter、Actuator、Sensor、Error Evaluator 和 Learning System。

15指标也要控制系统化

如果我们用控制系统视角评价 Agent，就不能只看任务成功率、耗时和成本。还应该关注是否稳定、是否过度修改、是否对扰动鲁棒、是否需要频繁人工介入。

控制指标Agent 指标解释
Settling Time从任务开始到稳定完成的时间不仅是生成速度，而是验证通过并可交付的速度
Overshoot过度修改、超出 scope 的程度Agent 是否为了完成任务改了太多不该改的东西
Steady-state Error最终仍未满足的验收项任务结束后仍存在的差距
Stability是否循环、震荡、反复改动能不能收敛
Robustness异常输入、工具失败、测试抖动下是否可控是否只在理想 demo 场景下有效
Control Efforttoken、工具调用、代码 diff、计算成本完成任务消耗了多少控制资源
Intervention Rate需要人类介入的频率和质量低价值打扰多不多，高风险上抛准不准
Constraint Violation Rate越权、超预算、越界修改次数是否安全可控

从这个角度看，“生成了多少行代码”不是核心指标。更好的指标是：

单位人类决策成本下，稳定交付了多少可验证、可维护的能力。

16给 Agent 系统设计者的实践清单

目标与状态

  • 是否把模糊需求转成可执行 Spec？
  • 是否有明确 Non-goals 和约束？
  • 是否知道当前系统状态？
  • 上下文是否有来源、时间和置信度？

反馈与误差

  • 是否有测试、日志、监控和 Review 传感器？
  • 是否能计算和 Spec 的差距？
  • 是否区分代码错误、环境扰动和测试抖动？
  • 是否存在快慢反馈分层？

控制与稳定

  • 是否小步执行、持续重新规划？
  • 是否限制最大 diff、最大步数和最大成本？
  • 连续失败后是否会降级或上抛？
  • 是否有 checkpoint 与 rollback？

多 Agent 与学习

  • 多 Agent 是否通过 DAG 协作，而不是群聊？
  • 是否有任务锁、文件锁和接口契约？
  • 经验沉淀是否有版本、适用范围和过期机制？
  • 错误知识是否可撤销、可审计？

17结语：从自主性走向可控自主性

很多关于 Agent 的讨论都在强调 autonomy，自主性。但控制系统视角提醒我们：自主性本身不是目标，可控自主性才是目标。

一个完全自主但不可控的 Agent，是风险；一个完全受控但不能自主的 Agent，只是脚本。理想的 Agent 应该在低风险、反馈充分、目标清晰的区域高度自主；在高风险、反馈不足、目标模糊的区域主动降级；在不可逆、跨边界、责任重大的地方上抛人类。

因此，成熟 Agent 平台应该同时具备两种系统能力：

  • Agent OS 负责运行边界：权限、资源、隔离、调度、审计、沙箱。
  • Agent Control System 负责目标收敛：Spec、状态估计、反馈、误差、稳定性、自适应。
真正成熟的 Agent 系统，不是让模型自由发挥，而是让智能在正确的边界内，通过持续反馈稳定逼近目标。

如果说操作系统视角让我们学会管理 Agent，那么控制系统视角让我们学会调教 Agent、观测 Agent、约束 Agent，并最终让 Agent 从“会生成”走向“会收敛”。
