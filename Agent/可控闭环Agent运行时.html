从提示词到运行时：构建可控闭环 Agent 的工程实践

围绕 Executor / Controller Loop、Task Contract、Error Vector、Output Gate、上下文隔离、Skill Sandbox 与虚拟工作区的一套实践设计。

主题：Agent Closed-loop Runtime · 多 Agent 协作 · Artifact Lifecycle · VFS / Sandbox · 信息流控制

核心观点：闭环 Agent 的价值不在于“多启动几个 subagent”，而在于把复杂任务放进一个有契约、有误差、有产物生命周期、有隔离边界、有输出门控的运行时里。模型只产生候选内容；Runtime 决定它被写到哪里、谁能看见、何时能成为最终输出。

一、为什么实践比理论更难

前面我们用控制系统类比 Agent：目标是 reference，当前产物是系统状态，评审结果是 error，执行 Agent 是推动状态变化的执行单元，控制 Agent 根据误差给出修正信号。这个理论框架是成立的，但工程实现时会遇到几个非常具体的问题。

  • 不是所有请求都值得进入闭环。“今天天气如何”和“帮我做一份 PPT”不是同一类任务。
  • 如果只是让一个 Agent 写、另一个 Agent 审，它仍然可能只是普通 subagent 协作。
  • “误差”“收敛”“震荡”在 Agent 任务里很难像经典控制系统那样写成传递函数。
  • 模型很难稳定地把正确内容写到正确位置。提示词约束输出位置非常脆弱。
  • 多 Agent 场景下，中间输出、私有草稿、错误假设很容易污染其他 Agent 的上下文。
  • 一旦引入 skill 脚本，脚本可以绕过工具 wrapper，直接通过文件系统读写。

因此，闭环 Agent Runtime 的核心不是“多 Agent 编排”，而是控制边界、信息流、产物状态和执行环境。

二、前置编排器：先决定是否需要闭环

一个成熟的 Agent Runtime 不应该把所有用户请求都扔进同一条长循环。第一步应该由前置编排器判断任务类型、复杂度、风险和是否需要产物。

模式适用任务执行方式
Mode 0：直接回答简单解释、常识问答、短文本改写单次模型输出，轻量校验
Mode 1：单步工具天气、汇率、日程、简单查询一次或少量工具调用，然后输出
Mode 2：有界闭环PPT、报告、代码、表格、文档整理Executor 生成候选产物，Controller 评估并迭代
Mode 3：受监督闭环生产变更、客户触达、权限、金钱、合规场景闭环 + 人类审批 + 强审计 + 回滚

这一步很重要，因为闭环有成本。简单任务如果也进入多轮评审，会显得慢、贵、啰嗦；复杂任务如果走开环，则很容易产生无法交付的半成品。

三、Task Contract：闭环的目标信号

如果没有 Task Contract，闭环评估就会变成“另一个 Agent 的主观意见”。控制 Agent 只有在知道目标、约束、验收标准和停止条件之后，才可能计算误差。

一个复杂任务进入闭环前，至少应该被编译成这样的契约：

{
  "goal": "制作一份关于闭环 Agent Runtime 的技术博客或 PPT",
  "output_type": "html_or_pptx",
  "audience": "技术团队、架构师、Agent 产品开发者",
  "constraints": [
    "不要使用不可渲染的 Mermaid",
    "不要出现 TODO 或占位符",
    "中间草稿不能进入最终输出"
  ],
  "acceptance_criteria": [
    "解释 Front Orchestrator 的任务分流",
    "解释 Executor / Controller / Agent Loop 的职责",
    "包含 Error Vector、Output Gate 和 VFS/Sandbox 设计",
    "有图示、有实践清单、有明确结论"
  ],
  "quality_rubric": {
    "structure": 0.85,
    "technical_depth": 0.85,
    "visual_clarity": 0.80
  },
  "max_iterations": 5,
  "stop_conditions": [
    "critical_errors == 0",
    "output_gate == pass",
    "或达到最大迭代次数并上抛用户"
  ]
}

Task Contract 的作用类似控制系统里的 reference signal。它不是给用户看的漂亮需求文档，而是给 Runtime 和 Controller 消费的目标向量。

实践原则：复杂任务不要直接让 Executor 执行。先把用户意图编译成 Contract，再让 Executor 生成候选产物，让 Controller 按 Contract 评估误差。

四、Executor / Controller / Agent Loop

闭环任务的主循环可以简化为：

Executor 生成 Candidate Artifact
        ↓
Controller 根据 Contract 评估 Error
        ↓
Agent Loop 判断 pass / revise / ask_user / fail
        ↓
如果 revise，则把 Delta Instruction 交回 Executor
        ↓
直到 Output Gate 通过或触发停止条件

这里要区分三层权力：

组件职责不应该做什么
Executor Agent生成候选产物，按上一轮修正指令做最小必要修改不直接写 final，不单方面宣布完成
Controller Agent产生结构化 ReviewReport、ErrorVector 和 DeltaInstruction默认不直接修改候选产物
Agent Loop Runtime维护状态、轮次、预算、收敛趋势和停止条件不生成大段业务内容
Output Gate做最终格式、证据、约束和交付检查不做创作，只做 promotion

五、它和普通 subagent 的差别

如果只是“一个 Agent 做，一个 Agent 审”，那和普通 subagent 没有本质区别。真正的差异在于运行协议。

维度普通 subagent闭环 Runtime
核心抽象主 Agent 委派任务TaskRun + Contract + Artifact Lifecycle
输入自然语言 promptTask Contract / Output Contract
评审自然语言建议结构化 ErrorReport + DeltaInstruction
循环主 Agent 临时决定Agent Loop Runtime 按状态机驱动
停止模糊，依赖模型判断pass / revise / ask_user / fail / max_iter
最终输出最后一个 Agent 的回复Output Gate promotion 后的 committed artifact
调试上下文里找原因可回放 Trace + Artifact Version + Review History

所以，subagent 是“谁来做”；闭环 Runtime 是“如何判断做得对不对、如何让结果持续逼近目标、何时停止、何时交付”。

六、Error Vector：如何度量误差、收敛与震荡

经典控制系统可以写出传递函数，Agent 系统很难做到这一点。像“帮我做一份 PPT”这样的任务，输出是复杂 artifact，目标又包含内容、结构、视觉、风格、文件可用性等多维标准。

因此，不要试图先定义一个完美标量误差，而应该定义 Error Vector。

误差维度含义PPT 示例
Existence产物是否存在没有生成 pptx 文件
Validity产物是否有效文件打不开、图片损坏
Completeness必需内容是否覆盖缺少 Output Gate 章节
Structure逻辑结构是否合理先讲实现再讲问题，顺序混乱
Content内容是否准确深入把 subagent 和闭环 Runtime 混为一谈
Visual视觉是否清晰文字太密、图示不清楚
Constraint是否违反硬约束用户要求不用 Mermaid，但页面仍有 Mermaid
Evidence是否有可用性证据没有检查文件是否能打开
Uncertainty哪些关键条件未知受众和演讲时长不明确

Runtime 可以基于 Error Vector 构造一个控制用进展函数：

V_t = 10 * critical_errors
    + 8  * blocking_errors
    + 6  * missing_requirements
    + 10 * constraint_violations
    + 5  * evidence_gap
    + 4  * uncertainty
    + 3  * quality_gap

这个函数不是“质量真理”，而是用来判断系统是否在收敛。它允许 Runtime 做出工程判断：

  • V_t 持续下降：继续迭代。
  • V_t 反复上下波动：可能震荡，降低修改幅度或冻结已通过部分。
  • V_t 连续上升：可能发散，触发恢复策略或上抛用户。
  • critical_errors == 0 且 Output Gate 通过：可以交付。

如何定义震荡和发散

震荡也可以被工程化定义，而不是停留在直觉上。

  • Error Reopen Rate：上一轮解决的问题下一轮又出现。
  • Sign Change：ΔV_t 的方向频繁反转。
  • Scope Thrashing：修改范围很大，但 error reduction 很小。
  • Review Flip-flop：同一验收项在 pass / fail 之间反复切换。

当这些指标超过阈值时，Agent Loop 应该停止自动“再试一次”，而应切换到 recovery：缩小修改范围、冻结已通过部分、回滚到最佳版本、换模型、换 Controller，或上抛用户。

七、输出路由与上下文隔离

多 Agent 系统中最常见的问题之一是：模型无法稳定地把内容输出到你想要的位置。即使提示词里说“请只写 final.md”，它也可能把中间草稿写进最终输出，或者把不该共享的内部分析放进下一个 Agent 的上下文。

正确原则是：

Agent 只生成候选 payload；Runtime 决定 destination、visibility、lifecycle 和是否能被其他 Agent 读取。

因此，不应该让 Agent 直接拥有任意路径写权限，例如 write_file(path, content)。更好的设计是让 Agent 提交结构化 envelope，由 Runtime 根据角色、阶段和权限路由到对应 channel。

{
  "message_type": "candidate_artifact",
  "producer": "executor_agent",
  "declared_visibility": "final",
  "payload": { "slides": "..." }
}

即使 Agent 声明 final，Runtime 也应该重新计算真实可见性：

{
  "declared_visibility": "final",
  "effective_visibility": "candidate",
  "reason": "executor_agent cannot write final_output"
}

Typed Channels，而不是共享聊天上下文

不要让所有 Agent 共享一条长 transcript。应该由 Runtime 管理 typed channels：

Channel写入者读取者是否给用户看
executor.privateExecutorExecutor否
artifact.candidateExecutorController / OutputGate否
review.reportControllerLoop / Executor通常否
runtime.traceRuntimeDebug / Audit否
output.finalOutputGateUser是

每个 Agent 的上下文都应该由 Context Builder 按需构造。Executor 不应该自动看到 Controller 的全部内部分析；Controller 也不需要看到 Executor 的所有私有 scratch。中间状态默认不可见，只有经过 promotion 才能进入共享上下文或最终输出。

八、Skill 脚本为什么会突破工具层约束

工具层可以控制显式工具调用。例如你可以包装 write_file，禁止写入 final 目录。但是 skill 一旦包含脚本，它就会绕过这个 wrapper。

# skill script 内部可以直接做文件操作
open("/workspace/output/result.md", "w").write("...")
open("../shared/context.md", "w").write("...")
open("/home/user/.ssh/id_rsa").read()

脚本不是在“听提示词”，而是在操作系统里执行代码。它会调用 open、read、write、rename、unlink 等系统接口。如果进程拥有普通宿主机权限，就可以绕过你对模型工具做的所有限制。

关键判断：只要允许执行脚本，就必须把它当作一个不可信进程来隔离。Prompt、tool wrapper、自然语言约定都不是安全边界。

九、VFS / Sandbox：给 Agent 一个虚拟工作区

你的虚拟空间类比非常准确：每个 Agent / Skill 看到的是一个全局的虚拟工作区，它以为自己在写某个路径，但真实写入会被 Runtime 映射到当前任务、当前角色、当前阶段的隔离目录。

这个系统可以分成四层：

层职责解决的问题
Sandbox文件系统、网络、进程和资源隔离脚本不能读写宿主机或其他任务
Workspace Mapper把虚拟路径映射到真实 artifact 目录/workspace/output 实际落到 candidate output
Artifact Lifecycle管理 draft / candidate / reviewed / approved / final文件名叫 final 不等于最终输出
Output Gate校验并提升候选产物只有通过 gate 的产物才能交付用户

推荐工作区结构

/workspace
  /input      # 只读：Task Contract、上游产物快照
  /resources  # 只读：skill 模板、示例、静态资源
  /repo       # 可选：只读 base + writable overlay
  /work       # 可写：中间工作目录
  /output     # 可写：候选产物目录
  /tmp        # 可写：临时目录

普通 Executor 永远不应该看到真实 final 目录。即使为了兼容脚本提供 /workspace/final，也应该映射到无害的 candidate alias，而不是最终交付位置。

十、实现建议：不要 hook read/write，优先用 OS 隔离

从实现上看，脚本最终确实会调用 open/read/write 等系统接口。但不建议你自己在应用层 hook read/write。原因是：

  • write() 只拿到文件描述符，路径权限通常已经在 open/openat 阶段决定。
  • LD_PRELOAD、语言级 monkey patch 只能做 demo，不适合作为安全边界。
  • 脚本可能是 Python、Node、Go、Rust、shell 或第三方 binary，应用层 hook 很容易被绕过。
  • 路径隔离、挂载视图、copy-on-write 本来就是操作系统文件系统机制擅长做的事。

更推荐的实现路径是：

不要做：
  hook read/write

优先做：
  每个 skill run 一个 sandbox / mount namespace
  只挂载允许读写的目录
  repo 使用 Overlay / copy-on-write
  output 映射到 candidate artifact
  final 不挂载或只读
  stdout/stderr 进入 trace，不进入其他 Agent 上下文
  执行结束后由 Runtime 收集 output 并进入 Output Gate

分层技术选择

方案适合场景说明
bind mount / mount namespace基础隔离让每个进程看到不同目录视图
OverlayFS / copy-on-write代码仓库、文档生成真实 base 只读，写入进入 overlay
tmpfs临时文件任务结束可直接销毁
seccomp / capabilities限制危险 syscall作为补充，不适合单独做路径翻译
Landlock / AppArmor / SELinux路径访问控制更硬的访问策略
FUSE VFS复杂动态映射可把路径映射到 artifact store，但实现复杂
Docker / Podman / nsjail / bubblewrapMVP 和工程化沙箱最快做出可用隔离环境
gVisor / Firecracker高风险、多租户、不可信 skill隔离更强，运维成本更高

代码仓库场景：用 Overlay 而不是直接改真实 repo

base repo:    readonly lowerdir
agent layer:  writable upperdir
merged view:  /workspace/repo

Agent 以为自己在改 /workspace/repo
真实写入进入 overlay
Runtime 计算 diff → scope check → tests → review → PR

这让 skill 脚本可以运行复杂重构，同时不会污染真实仓库。失败时直接丢弃 overlay；成功时把 diff 打包成 candidate patch。

十一、Runtime 模块化设计

如果基于这套哲学开发一个 Agent Runtime，最小但有价值的系统应该包含这些模块。

模块职责
Task Router判断任务模式：直接回答、单步工具、有界闭环、受监督闭环
Contract Compiler把用户意图编译成 Task Contract、Review Rubric、Output Contract
Agent Loop Runtime管理迭代、状态、预算、停止条件、收敛趋势和上抛策略
Executor Runtime根据 Contract 和 DeltaInstruction 生成候选产物
Controller Runtime生成 ReviewReport、ErrorVector 和结构化修正指令
Channel Registry定义 typed channel、读写权限、可见性和生命周期
Context Builder按角色和阶段构造上下文，阻止中间状态污染
Artifact Store管理 candidate / reviewed / approved / final 版本
Sandbox Manager为 skill 脚本创建隔离进程和虚拟工作区
Workspace Mapper把虚拟路径映射到真实 artifact / overlay / tmp 目录
Output Router校验 Agent 输出 envelope，并路由到正确 channel
Output Gate最终格式、证据、约束和风险检查；执行 promotion
Trace Store记录所有运行事件，用于 replay、debug、audit 和评估

十二、MVP 路线与落地清单

如果要把这套设计落地，我建议不要一开始做通用 Agent 平台，而是先做一个 artifact-oriented closed-loop runtime。例如先支持技术博客、PPT、报告或代码 PR 草案。

第一阶段：最小闭环

  • Task Router：判断是否进入闭环。
  • Contract Compiler：生成目标、约束、验收标准和最大迭代次数。
  • Executor：生成 candidate artifact。
  • Controller：输出结构化 ReviewReport。
  • Loop Runtime：根据 review 决定继续、停止或交付。
  • Output Gate：检查文件、格式、占位符、blocking issue。

第二阶段：产物和上下文治理

  • 引入 typed channels，而不是共享 transcript。
  • 引入 Artifact Store，支持版本化和 promotion。
  • 引入 Context Builder，按角色构造上下文。
  • 引入 Trace Store，所有中间日志默认只进入 trace，不进入下游上下文。

第三阶段：Skill Sandbox

  • 每次 skill run 创建独立 workspace。
  • /input 和 /resources 只读，/output 只是 candidate。
  • 代码仓库使用 overlay / copy-on-write。
  • 禁止默认网络访问，过滤 env 和 secrets。
  • stdout/stderr 只进 trace，产物由 Output Collector 收集。

第四阶段：量化控制

  • 把评审结果转成 Error Vector。
  • 记录每轮 V_t、成本、修改范围和用户反馈。
  • 检测震荡、发散和低收益迭代。
  • 积累 loop traces，估计不同控制动作的成功率、成本和回归概率。

结语：闭环 Agent 的本质是可控交付

Agent Runtime 的目标不是让模型“更自由”，而是让模型在正确的边界内持续产生可交付结果。

这需要把几个容易混在一起的概念彻底拆开：

  • 模型生成的是候选内容，不是最终输出。
  • Executor 负责生成候选状态，Controller 负责估计误差。
  • Agent Loop Runtime 负责迭代、收敛、停止和上抛。
  • Output Gate 负责把 candidate 提升为 committed output。
  • Context Builder 负责谁能看到什么。
  • Sandbox 和 VFS 负责脚本实际能读写哪里。
  • Artifact Lifecycle 负责状态如何从 draft 走到 final。

一句话总结：闭环 Agent Runtime 不是把两个 subagent 接起来，而是建立一套从意图到契约、从候选产物到误差信号、从虚拟工作区到最终输出的受控交付系统。

当 Runtime 能回答这些问题时，Agent 系统才真正进入工程化阶段：

这个任务是否需要闭环？
当前产物和目标差多少？
下一轮应该局部修复还是重做？
系统是在收敛、震荡还是发散？
谁能写 candidate，谁能写 review，谁能写 final？
中间输出是否会污染其他 Agent？
skill 脚本能否越权读写？
最终产物为什么可以交付？

这些问题的答案，不应该藏在提示词里，而应该成为 Runtime 的一等机制。
