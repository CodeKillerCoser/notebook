成熟 Computer Use 系统如何落地：动作协议、OS 接口与钉钉发送消息全流程

Computer use 不是“让模型随便点屏幕”。真正可产品化的实现，应该是一套能看屏幕、读 UI 树、管理窗口、启动应用、执行输入、验证结果、拦截高风险动作并可审计回放的 GUI 自动化运行时。本文把前面的讨论整理成一份工程化设计文档，并用“Windows 上打开钉钉并给 xxx 发送消息”作为贯穿案例。

1一、Computer use 到底是什么

Computer use 的核心是一个闭环：模型观察屏幕，输出下一步 UI 动作；执行层把动作转换成真实的鼠标、键盘、窗口、剪贴板或无障碍接口调用；执行后再截图和读取系统状态，把新的观察结果交回模型，直到任务完成。OpenAI 的官方 Computer use 文档把这个循环描述为：启用 computer 工具发送任务，检查返回的 computer_call，顺序执行 actions[]，捕获更新后的屏幕作为 computer_call_output 回传，重复直到模型不再返回 computer call。[1]

因此，它不是传统意义上的 API 调用，也不是简单的 RPA 脚本。它更像“带视觉的控制器”与“受控桌面运行时”的组合。OpenAI 对 CUA 的介绍也强调，Computer-Using Agent 通过视觉和推理能力与图形界面交互，面对的是人类在屏幕上看到的按钮、菜单和文本框。[2]

一句话定义：Computer use = 视觉理解模型 + 结构化动作协议 + OS/浏览器执行器 + 反馈截图 + 安全策略 + 审计日志。

在 demo 里，你可以只做“截图 → 点坐标 → 再截图”。但在真实业务里，这远远不够。比如“打开钉钉并发送消息给 xxx”这个任务，模型需要回答一连串非视觉问题：钉钉是否安装？是否已经启动？窗口是否最小化？是否需要登录？搜索到多个同名联系人怎么办？发送消息前是否需要用户确认？发送后如何验证成功？这些都不是单纯靠点击坐标能可靠解决的。

2二、成熟系统的运行时架构

成熟的 computer use 系统应该被设计成一个运行时，而不是一个脚本。它至少包含六层：模型控制器、编排器、观察层、执行层、OS 适配层、安全治理层。

图 1：成熟 Computer Use Runtime 的核心闭环
不要让模型直接拥有电脑。模型只提出动作；Runtime 负责验证、执行、观察和审计。

这套架构有一个重要分工：

模型负责推理，Runtime 负责治理；模型提出动作，Kernel 决定是否允许动作。

成熟系统要避免两个极端：一是所有能力都交给模型自由点击，导致不可控；二是所有流程都写死成脚本，失去 computer use 的自适应能力。更好的方式是：结构化能力优先，视觉能力兜底；程序化接口优先，坐标点击兜底；低风险动作自动执行，高风险动作上抛确认。

3三、动作协议：不要只暴露坐标点击

底层动作当然需要存在。OpenAI Computer use 的动作集合包括点击、双击、滚动、输入、等待、按键、拖拽、移动和截图等 UI 交互动作。[1] 但成熟实现不应只向模型暴露这些原始动作，而应该分层设计动作协议。

1. 人类输入模拟层

observe_screen()
screenshot()
move(x, y)
click(x, y, button="left")
double_click(x, y)
right_click(x, y)
drag(from_x, from_y, to_x, to_y)
scroll(x, y, delta_x, delta_y)
keypress(keys)
type_text(text)
wait(ms)

这一层模拟人类鼠标键盘，适合做兜底，但不稳定。UI 改版、窗口缩放、DPI、遮挡、多显示器、滚动位置变化都会影响坐标。

2. 控件语义层

find_element(role="Edit", name_contains="搜索")
click_element(element_id)
set_value(element_id, text)
invoke(element_id)
select(element_id)
expand(element_id)
collapse(element_id)
scroll_into_view(element_id)
get_text(element_id)

这一层基于无障碍树或 UI 自动化树。Windows 上通常使用 Microsoft UI Automation。UI Automation 为桌面 UI 元素提供程序化访问，自动化脚本也可以通过它与 UI 交互。[3] UI Automation 的 control patterns 则把控件能力抽象成接口，例如 Invoke、Value、Selection、Scroll、Text、Window 等。[4]

3. 应用与窗口层

list_apps(query)
find_app(name_aliases)
is_app_installed(app_ref)
launch_app(app_ref)
list_processes(query)
list_windows(query)
get_active_window()
focus_window(window_id)
restore_window(window_id)
wait_for_window(predicate, timeout)
close_window(window_id)

这层是很多 demo 缺失、但产品化必须补上的能力。以“打开钉钉”为例，模型不应该通过视觉去猜开始菜单里有没有钉钉，而应该先通过系统清单、进程和窗口状态判断。

4. 安全与人工确认层

policy_check(action)
request_user_confirmation(summary)
handoff_to_user(reason)
redact_sensitive_observation()
record_audit_event(step, observation, action, result)

发消息、发邮件、付款、删除、修改权限、提交表单、上传文件都属于外部影响动作。它们不能和“滚动页面”同等对待。Runtime 必须先分类风险，再决定自动执行、拒绝、或请求用户确认。

动作层级典型动作优先级主要风险
系统动作查安装、启动应用、聚焦窗口最高误启动、权限不足
控件动作find_element、invoke、set_value高UI 树不完整、控件命名不稳定
剪贴板/输入动作paste_text、keypress中焦点错误、剪贴板污染、IME 问题
坐标动作click、drag、scroll兜底DPI、遮挡、滚动位置、误点
安全动作confirm、handoff、audit横切确认缺失、审计缺失、越权

4四、观察层：截图、UI 树与系统状态

模型要做出正确决策，首先需要完整、稳定、低噪音的 observation。成熟系统的一次观察不应该只有一张截图，而应该包含：

{
  "screenshot": {
    "width": 1920,
    "height": 1080,
    "scale": 1.25,
    "monitors": [...]
  },
  "active_window": {
    "id": "win_8",
    "title": "钉钉",
    "process": "DingTalk.exe",
    "rect": [0, 0, 1280, 900]
  },
  "windows": [...],
  "accessibility_tree": [...],
  "cursor": { "x": 420, "y": 88 },
  "clipboard_state": "text/plain available, content hidden",
  "security_context": {
    "logged_in_app": "unknown",
    "sensitive_regions_redacted": true
  }
}

尤其要注意三点。

1. 坐标系必须统一

截图坐标、窗口坐标、UIA bounding rectangle、鼠标坐标、多显示器坐标、DPI 缩放坐标必须统一。否则模型看到的是 1280×720，但执行器点的是 2560×1440，必然误点。

2. UI 树和截图要互相校验

UI Automation 或 AX/AT-SPI 可能缺控件，截图可能看不清文本。成熟系统应该同时给模型视觉和结构化树：有语义时走语义控件；语义树缺失时用视觉兜底。

3. 观察结果要做安全处理

密码框、验证码、token、邮箱、手机号、身份证号、精确地址等敏感内容不应直接暴露给模型或日志。观察层应该具备识别与遮罩能力，并在必要时切到用户接管模式。

5五、操作系统接口映射

一个跨平台 computer use runtime 应该定义统一接口，再为不同 OS 实现 adapter。统一动作是 launch_app、focus_window、find_element，底层则分别落到 Win32/UIA、macOS AX/CGEvent、Linux AT-SPI/XDG/Wayland/X11。

macOS 的 Accessibility API 中，AXUIElement 表示可访问对象，并提供 UI 对象的位置信息和属性；Quartz 的 CGEvent 则可表示鼠标或键盘事件。[10] Linux 桌面环境更分裂，AT-SPI2 是通过 D-Bus 暴露无障碍信息的协议，freedesktop 的 Desktop Entry Specification 则定义了应用入口和启动信息的标准格式。[11]

6六、Windows 适配层设计

Windows 是企业桌面自动化里最常见、也最复杂的目标环境。一个成熟 Windows adapter 至少要对接以下能力。

1. 应用发现

推荐组合多个来源：

PowerShell Get-StartApps
Start Menu .lnk 扫描
HKCU / HKLM Uninstall 注册表项
常见安装目录
正在运行的进程
企业软件分发清单

Get-StartApps 可以获取当前用户安装应用的名称和 AppID。[7] 但它不一定覆盖所有传统 Win32 软件，所以还要补充 Start Menu、注册表和进程扫描。

2. 应用启动

启动应用时，Runtime 应该先拿到一个结构化 app_ref，而不是让模型拼命令行：

{
  "display_name": "钉钉",
  "aliases": ["DingTalk", "钉钉"],
  "start_app_id": "...",
  "shortcut_path": "C:\\Users\\...\\Start Menu\\Programs\\钉钉.lnk",
  "exe_path": "C:\\Program Files (x86)\\DingDing\\DingtalkLauncher.exe",
  "confidence": 0.91
}

ShellExecute / ShellExecuteEx 适合按 Shell 语义打开文件、文件夹、快捷方式或应用；CreateProcessW 则用于创建新进程及其主线程。[8]

3. 窗口管理

窗口管理需要提供：

EnumWindows → 枚举顶层窗口
GetWindowText → 读窗口标题
GetWindowThreadProcessId → 绑定窗口与进程
IsWindowVisible → 判断是否可见
ShowWindow(SW_RESTORE) → 恢复最小化窗口
SetForegroundWindow → 请求前台激活

EnumWindows 会枚举屏幕上的顶层窗口。[9] SetForegroundWindow 会把创建指定窗口的线程带到前台并让键盘输入指向该窗口，但 Windows 对抢焦点有系统限制，不是任何进程都能随时成功。[9] ShowWindow 则用于控制窗口显示状态，例如恢复最小化窗口。[9]

4. 控件读取与操作

Windows 上的控件语义层应优先使用 Microsoft UI Automation。它可以读取 Name、ControlType、AutomationId、BoundingRectangle、IsEnabled、IsOffscreen、ProcessId、SupportedPatterns 等字段，并通过 InvokePattern、ValuePattern、SelectionPattern、ScrollPattern 等执行语义动作。[3][4]

5. 输入执行

键盘鼠标兜底通常落到 SendInput。Microsoft 文档说明，SendInput 会把输入事件串行插入键盘或鼠标输入流，同时也受 UIPI 限制：应用只能向同等或更低完整性级别的应用注入输入。[5] 因此，执行层应该优先级如下：

UIA set_value / invoke
→ clipboard paste
→ SendInput keypress / mouse click
→ 视觉坐标兜底
→ 用户接管

6. 截图

现代 Windows 可以使用 Windows.Graphics.Capture 获取显示器或应用窗口帧，用于截图、视频流或交互场景。[6] 对 computer use 来说，截图模块要同时处理 DPI、多显示器、窗口遮挡、鼠标光标、UAC 安全桌面和敏感区域遮罩。

7七、闭环执行与状态机

成熟 computer use 不能只是“模型返回动作就执行”。它应该有显式状态机、最大步数、超时、稳定性检测、结果验证和失败恢复。

图 2：Computer Use 执行状态机
每一步都要有可观察的前置条件、动作、后置验证和失败路径。

状态机的关键不是名字，而是保证每次动作前后都能回答三个问题：

  1. 当前状态是否足以执行这个动作？比如是否确认了目标联系人？
  2. 这个动作是否允许自动执行？比如发送消息是否需要确认？
  3. 执行后如何验证结果？比如消息是否真的出现在聊天气泡中？

8八、案例：Windows 上打开钉钉并发送消息给 xxx

钉钉官方下载安装页显示客户端支持 Windows、Mac、iOS、Android 等平台。[12] 下面假设用户任务是：

打开钉钉，给 xxx 发送消息：“今天下午三点开会”

Step 0：任务解析与风险分类

{
  "app": "钉钉",
  "recipient": "xxx",
  "message": "今天下午三点开会",
  "action": "send_message",
  "risk": "external_communication",
  "requires_confirmation": true
}

发送消息是外部通信，默认需要用户确认。除非系统有明确白名单，例如测试机器人、测试群、低风险通知模板，否则不应自动点击发送。

Step 1：发现钉钉是否安装

Runtime 执行：

find_app(["钉钉", "DingTalk", "Ding Ding", "DingDing"])

内部策略：

  1. 用 Get-StartApps *钉钉* 和 Get-StartApps *DingTalk* 查当前用户 Start apps。
  2. 扫描 Start Menu 快捷方式。
  3. 检查 HKCU/HKLM Uninstall 注册表项。
  4. 检查当前进程和窗口。
  5. 必要时查企业软件分发清单。

如果没有安装，不要自动下载安装。正确行为是告诉用户未找到钉钉，并询问是否打开官方下载页或让用户安装。

Step 2：判断钉钉是否已经启动

系统执行：

list_processes(query=["DingTalk", "DingDing", "钉钉"])
list_windows(query=["钉钉", "DingTalk"])
状态判断动作
进程和主窗口都存在有 DingTalk 进程，有可见顶层窗口恢复窗口、聚焦、观察主界面
进程存在但无主窗口可能缩到托盘或后台二次 launch 唤起主窗口；失败再处理托盘或用户接管
未运行但已安装找到 app_ref，无进程ShellExecuteEx 或 CreateProcessW 启动
出现登录或验证界面观察到二维码、验证码、密码框用户接管，不绕过认证

Step 3：等待主窗口稳定

启动后不要马上点击。应等待：

window.visible == true
active_window.process matches DingTalk
screenshot stable for 500-1000ms
UI tree contains main window or search/message entry

稳定后返回综合 observation：当前窗口标题、可见文本、UIA 树候选控件、截图、DPI、窗口位置。

Step 4：定位搜索入口

优先走语义控件：

search_box = find_element(
  role="Edit",
  name_contains=["搜索", "search"],
  within="DingTalk main window"
)

如果找到了并支持 ValuePattern：

set_value(search_box, "xxx")

如果不支持 ValuePattern，则：

click_element(search_box)
paste_text("xxx")

如果 UI 树缺失，退回视觉方案：OCR 找“搜索”文本附近输入框，或让模型根据截图点击搜索区域，再粘贴联系人名。

Step 5：搜索并选择联系人

输入联系人后等待搜索结果：

wait_until(
  UI tree contains result text "xxx"
  OR screenshot/OCR contains "xxx"
  OR timeout
)

成熟系统不能只点第一个结果。它应该构建候选列表：

[
  {
    "element_id": "el_result_1",
    "display_name": "xxx",
    "subtitle": "产品部 / 公司内部",
    "type": "person",
    "confidence": 0.92
  },
  {
    "element_id": "el_result_2",
    "display_name": "xxx 项目群",
    "type": "group",
    "confidence": 0.66
  }
]

如果只有一个高置信候选，打开它；如果有多个候选，必须让用户选择。对 IM 场景来说，发错人比任务失败更严重。

Step 6：验证已经进入正确会话

选择结果后不能立即发消息，应先验证：

  • 聊天窗口标题或顶部文本包含目标联系人。
  • 最近会话选中项与目标一致。
  • 联系人类型符合预期：个人、群聊、机器人或外部联系人。
  • 不存在“搜索结果仍未选择”“会话加载中”“网络错误”等状态。

Step 7：定位消息输入框并写入草稿

消息输入框可能是富文本控件，UIA ValuePattern 不一定可用。推荐策略：

input_box = find_element(role="Edit", name_contains=["消息", "输入", "请输入"])
click_or_focus(input_box)
paste_text("今天下午三点开会")
verify_draft_text("今天下午三点开会")

中文文本优先粘贴，而不是逐字模拟键盘输入。这样可以减少 IME 候选词、键盘布局、快捷键冲突带来的不确定性。

Step 8：发送前确认

必须停一下

发送消息属于外部影响动作。默认应在真正点击“发送”前展示确认：

即将通过钉钉向「xxx」发送：

今天下午三点开会

是否确认发送？

Step 9：执行发送

用户确认后，优先找“发送”按钮并 invoke：

send_button = find_element(role="Button", name_contains=["发送", "Send"])
invoke(send_button)

不要默认按 Enter，因为 IM 软件可能配置为 Enter 发送、Ctrl+Enter 发送，或者 Enter 换行。找不到发送按钮时，才根据界面提示或配置使用快捷键。

Step 10：发送后验证

发送后再次观察：

  • 输入框是否清空。
  • 最后一条消息气泡是否包含目标文本。
  • 气泡旁是否有失败标记、红色感叹号或网络错误。
  • 当前会话对象是否仍是 xxx。

只有这些验证通过，才返回“已发送”。

这不是“打开应用 → 点搜索 → 按回车”的线性脚本，而是一个带验证和安全闸门的状态机。

9九、关键分支与失败恢复

成熟系统的价值主要体现在失败路径，而不是 happy path。

1. 钉钉未安装

检测：Start apps、快捷方式、注册表、进程均未找到。
动作：不自动下载安装。
输出：本机似乎未安装钉钉，是否打开官方下载安装页？

2. 钉钉已启动但缩到托盘

检测：DingTalk 进程存在，但没有可见主窗口。
动作：再次 launch_app，很多 IM 应用会唤起主窗口。
失败：尝试托盘区 UIA / 视觉回退；仍失败则用户接管。

3. 需要登录、扫码或验证码

检测：二维码、验证码、密码框、二次认证页面。
动作：handoff_to_user("钉钉需要登录或验证，请先完成登录")。
禁止：模型代填、绕过、读取或保存敏感凭据。

4. 搜索到多个 xxx

检测：候选列表中存在多个同名个人、群聊、机器人或外部联系人。
动作：展示候选，让用户选择。
禁止：默认选第一个。

5. 找不到发送按钮

检测：UIA 无 Button("发送")，截图中也没有明确按钮。
动作：读取界面提示或配置；必要时询问用户发送快捷键。
禁止：默认 Enter 一定是发送。

6. 发送后失败

检测：最后气泡旁有红色感叹号、网络错误、输入框未清空。
动作：返回失败证据，允许用户确认后重试。
禁止：没有验证就声称已发送。

10十、安全治理与人工确认

Computer use 的风险高于普通聊天，因为它能操作真实账号、真实文件和真实业务系统。因此，安全治理不是附加项，而是 Runtime 的核心能力。

图 4：动作输出端的安全过滤器
所有外部动作都应该像受控 syscall 一样进入统一安全过滤器。

1. 最小权限

Agent 只应访问完成任务所需的应用、网站、文件和网络范围。不要让一个通用 Agent 默认继承用户全部权限。

2. 高风险动作确认

发送消息、发送邮件、提交表单、付款、删除文件、修改权限、上传敏感文件，都应该要求用户确认。确认界面必须展示对象、内容、后果和可撤销性。

3. Prompt injection 防护

屏幕上的网页、邮件、聊天消息、PDF、文档和工具输出都应被视为不可信输入。第三方内容不能自动获得系统指令级别的权限，更不能被当作用户授权。

4. 审计与回放

每一步应记录：时间、任务 ID、观察摘要、动作、策略判断、执行结果、截图哈希、用户确认记录。这样才能复现错误、追责和回滚。

11十一、实现建议与伪代码

实现时推荐把系统拆成四个核心接口：Observer、Planner、Policy、Executor。

class ComputerUseRuntime:
    def run(self, task):
        state = self.parse_task(task)

        for step in range(MAX_STEPS):
            obs = self.observer.observe()
            action_plan = self.planner.next_action(state, obs)

            if action_plan.is_done:
                return self.verifier.final_result(state, obs)

            decision = self.policy.check(action_plan, state, obs)
            if decision.requires_user_confirmation:
                if not self.ask_user(decision.summary):
                    return Cancelled()

            if decision.denied:
                return Blocked(decision.reason)

            result = self.executor.execute(action_plan)
            self.audit.record(step, obs, action_plan, decision, result)

            if not result.ok:
                state = self.recovery.update(state, result)
                continue

            state = self.reducer.update(state, obs, action_plan, result)

        return Failed("任务未在最大步数内完成")

钉钉发送消息的业务伪代码可以写成：

def send_dingtalk_message(recipient, message):
    app = find_app(["钉钉", "DingTalk", "DingDing"])
    if not app:
        return ask_user("未找到钉钉。是否打开官方下载安装页？")

    win = find_window_by_app(app)
    if not win:
        launch_app(app)
        win = wait_for_window(app, timeout=30)

    if not win:
        return handoff_to_user("钉钉无法启动或没有出现主窗口")

    restore_window(win)
    focus_window(win)

    obs = observe(include=["screenshot", "accessibility_tree", "active_window"])
    if detect_login_required(obs):
        return handoff_to_user("钉钉需要登录或验证，请先完成登录")

    search = find_element(role="Edit", name_contains=["搜索", "search"], within=win)
    if search:
        set_or_paste(search, recipient)
    else:
        box = locate_by_ocr_or_vision("搜索")
        click(box.center)
        paste_text(recipient)

    results = wait_search_results(recipient)
    chosen = disambiguate_results(results, recipient)
    if chosen.needs_user_choice:
        return ask_user_to_choose(chosen.candidates)

    invoke_or_click(chosen.element)

    if not verify_chat_opened(recipient):
        return fail("无法确认当前会话对象是目标联系人")

    input_box = find_message_input()
    click_or_focus(input_box)
    paste_text(message)

    if not verify_draft_text(message):
        return fail("消息草稿内容未正确写入")

    confirmed = ask_user_confirmation(
        f"将通过钉钉向「{recipient}」发送：\n\n{message}"
    )
    if not confirmed:
        return cancel()

    send_button = find_element(role="Button", name_contains=["发送", "Send"])
    if send_button:
        invoke_or_click(send_button)
    else:
        keypress(["ENTER"])

    if verify_message_sent(message):
        return success()
    return fail("消息可能未发送成功，请检查钉钉窗口")

几个实现细节

  • 动作要幂等：重复执行 focus_window 应该安全；重复执行“发送”则不安全，必须禁止自动重试。
  • 等待要看状态：不要固定 sleep 3 秒，而是等待窗口出现、UI 树稳定、截图变化停止或文本出现。
  • 坐标要可解释：坐标动作必须带理由和目标区域，例如“点击搜索框中心”，而不是裸坐标。
  • 错误要可恢复：每个步骤都要知道失败后是重试、换策略、询问用户，还是中止。
  • 版本要可观测：记录应用版本、窗口标题、DPI、屏幕分辨率和 OS 版本，方便定位兼容性问题。

12十二、实践清单

应该做

  • 把 computer use 当作 Runtime，而不是脚本。
  • 优先用系统接口和 UI 树，坐标点击只做兜底。
  • 每个动作都有前置条件、后置验证和失败分支。
  • 发送、删除、付款、提交等动作必须经过策略检查。
  • 记录可审计日志和关键截图哈希。
  • 把失败案例沉淀成回归测试。

不要做

  • 不要让模型直接自由控制用户真实桌面。
  • 不要只靠截图和坐标完成复杂任务。
  • 不要默认第一个搜索结果就是目标联系人。
  • 不要默认 Enter 一定是发送。
  • 不要绕过登录、验证码或二次认证。
  • 不要没有验证就声称任务完成。

最终结论：成熟的 computer use 系统，真正的护城河不在“模型能不能看懂按钮”，而在 Runtime 能不能把不确定的模型行为，约束在可观察、可验证、可恢复、可审计、可授权的执行边界内。
