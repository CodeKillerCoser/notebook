UI 设计系统化完全指南：从“品味差”到“有规则地做出专业界面”

这份报告不是教你“把界面装饰得好看”，而是把 UI 拆成可判断、可复用、可实现的系统：信息架构、视觉层级、字体、间距、颜色、组件、状态、可访问性、响应式、数据展示、设计评审与工程落地。

生成日期：2026-07-06 适合：程序员 / 独立开发者 / 产品工程师 形式：图文报告 + checklist + 代码 token 可放入 notebook 仓库

先给你一个总判断

UI 设计的专业感，主要不是来自“天生审美”，而是来自 约束系统。你写程序不会每个页面都随机起变量名、随机选架构；UI 也一样。字体、字号、颜色、间距、圆角、按钮、表格、表单、反馈状态，都应该来自一套规则。

清楚
用户第一眼知道这是什么页面、最重要的信息在哪里、下一步能做什么。
一致
相同的东西长得相同，相同的操作行为一致，相同的状态表达一致。
可用
能读、能点、能撤销、能理解错误；弱网、空数据、权限不足时也不崩。

01UI 的全景地图：你到底需要学哪些角落

UI 不是只有视觉。一个完整界面至少包含九层：目标、信息、布局、视觉、组件、交互、状态、内容、工程实现。你可以把它理解成前端架构，只不过这里的模块是视觉与行为。

层级你要解决的问题常见错误专业做法
产品目标用户为什么来这个页面？先画界面，后想目标先定义主任务、主信息、主操作
信息架构内容如何分组、排序、命名？把所有功能平铺按用户任务分组，减少认知负担
布局元素放在哪里？到处居中、对齐线混乱网格、对齐线、容器、留白
视觉系统字体、颜色、间距、圆角如何统一？每个地方临时决定设计 token：颜色角色、字号阶梯、间距刻度
组件按钮、表单、表格、卡片如何复用？同一组件多种样式组件规范 + 状态规范 + 变体规范
交互用户怎么操作？操作后发生什么？没有 hover、loading、focus、反馈默认、悬停、按下、聚焦、禁用、加载、成功、失败全覆盖
内容文案界面文字是否可理解？“Error”“提交失败”写清原因、影响、下一步
可访问性不同能力、设备、环境下是否能用？浅灰小字、只靠颜色、点击区太小对比度、键盘、焦点、语义、触控目标
工程实现设计如何稳定落地？CSS 到处硬编码token、组件 API、响应式规则、视觉回归检查
程序员学习 UI 的最佳路线：不要先追潮流，不要先学 Figma 特效。先掌握可复用的基础规则：字号阶梯、8px 间距、颜色角色、按钮层级、表单状态、表格密度、空/错/加载状态。然后再学风格。

03七条底层原则：所有 UI 决策都从这里出发

1. 清晰性优先

用户不应该猜页面是什么、按钮会做什么、错误意味着什么。清楚永远比炫酷重要。

2. 层级明确

一个页面必须有第一重点、第二重点、第三重点。没有层级就没有阅读路径。

3. 一致性降低成本

同样的动作、状态、组件，在不同页面应该长得一样。每一次例外都会增加用户学习成本。

4. 反馈必须及时

点击、提交、加载、失败、保存成功都要有反馈。无反馈会让用户重复点击或不信任系统。

5. 容错比美观更重要

撤销、确认、错误提示、禁用状态、校验规则、失败恢复，是专业产品和玩具 Demo 的分界线。

6. 视觉为任务服务

颜色、阴影、动效不是装饰，而是告诉用户：什么重要、什么可点、什么危险、什么已完成。

7. 系统胜过灵感

专业 UI 不是每次灵光一闪，而是把字体、颜色、间距、组件状态都放进一个系统。

8. 默认状态之外才见功力

空状态、加载态、错误态、无权限、弱网、长文本、极端数据，才是真实产品最容易露怯的地方。

字体不是“选一个好看的字体”这么简单。UI 排版要解决的是阅读速度、层级、密度、平台一致性和中英文混排。

4.2 程序员可直接使用的字号系统

12px辅助信息、表格说明、时间戳、标签。不要用于大量正文。
14px后台正文、表格内容、按钮、表单 label。Web 工具型产品常用。
16px正文、移动端输入、阅读型页面基础字号。移动端正文尽量从 16px 起。
18/20px区块标题、卡片标题、弹窗标题。
24/28px页面标题、重要模块标题。
32/40/48px营销页 Hero、重要数字、数据大屏，不要滥用。

中英文混排中文、英文、数字挤在一起数字可用 tabular-nums；金额、百分比要统一格式
文本长度一行过长，读起来疲劳阅读正文宽度控制在 640～760px 左右；后台按内容需要扩展

05间距、网格与对齐：专业感的骨架

很多页面看起来“不高级”，真正原因不是配色，而是没有对齐线、没有分组逻辑、没有间距系统。UI 的留白不是空白，而是关系。

坏例子：随机间距，没有对齐线
标题
输入框
说明文字
按钮

问题：元素左右边缘不一致，组内组外距离差不多，按钮位置飘。

好例子：8px 系统 + 共同对齐线
页面标题 / 24px
描述文字 / 与标题同组，距离 8px
表单字段 / 区块间距 24px
主按钮 / 与表单间距 24px

改进：所有元素共享左边界，组内近、组间远，页面有稳定骨架。

5.1 4/8px 间距系统

481216243240486480

新手先不要写 margin: 7px、padding: 19px。除非你有明确原因，否则所有空间都从这组数字里取。

关系推荐间距解释
图标与文字6 / 8px属于同一按钮或同一标签，距离小
Label 与输入框6 / 8px同一字段内部关系
表单字段之间16 / 20 / 24px同类字段之间，需清楚分隔
卡片内边距16 / 24 / 32px内容密度越低，padding 越大
区块之间32 / 48 / 64px大信息组之间需要明显留白
页面边距移动端 16/20px；桌面 24/32/48px不同设备要有不同安全边距

5.2 栅格、容器、密度

后台、管理系统、Dashboard 常用 12 栅格或 CSS Grid。你不必每次显式画 12 列，但要知道：内容不应该随意漂浮，应该落在一套列宽与间距上。

紧凑密度

适合表格、日志、运维工具。效率高，但对新手用户不友好。

标准密度

适合大多数 Web 产品，是默认选择。

宽松密度

适合营销页、移动端、复杂表单、首次使用场景。

06配色：不要选颜色，要定义颜色角色

专业 UI 的颜色不是“我喜欢蓝色/紫色”，而是角色：背景、表面、文字、边框、主操作、危险、成功、警告、信息、选中、悬停、禁用。

6.2 对比度与色弱友好

WCAG 2.2 的 AA 标准要求普通文本和背景至少达到 4.5:1 对比度，大字号文本至少 3:1。实际做产品时，不要用“看着高级”的浅灰小字牺牲可读性。
错误为什么错修正
浅灰正文 #aaa 放白底看起来轻，但读不清正文用深灰；辅助文字也要可读
只用红色表示错误色弱用户可能分辨困难红色 + 图标 + 错误文案
按钮背景浅、文字也浅点击前就看不清主按钮文字与背景高对比
图表用十几种随机颜色难比较，也不适合色弱少色、同色系、必要时用线型/图案/标签

07圆角、边框、阴影、层级：让界面有空间秩序

圆角、边框、阴影不是为了“更漂亮”，而是表达层级与可交互性。普通内容容器不一定需要阴影；浮层、菜单、弹窗更适合使用阴影。

圆角

6px 小标签，8px 按钮/输入框，12px 卡片，16px 弹窗，999px 胶囊/头像。

边框

普通卡片优先用 1px #e5e7eb。边框太黑会让界面像 Excel。

阴影

用来表达浮层：Dropdown、Popover、Modal、Toast。不要每个按钮都加大阴影。

层级典型元素视觉表达
Base页面背景白色或浅灰，无阴影
Surface卡片、列表区域白底 + 轻边框
RaisedDropdown、Popover白底 + 中等阴影
OverlayModal、Drawer遮罩 + 强阴影 + 更大圆角
SystemToast、全局提示固定位置，短时出现，层级最高

08按钮：操作优先级的视觉语言

按钮设计的核心不是颜色，而是优先级。一个区域通常只有一个 Primary。危险操作不应该和主操作并列抢眼，除非当前任务本身就是危险操作确认。

从左到右：主操作、次操作、弱操作、危险操作、禁用/加载态。不要把所有按钮都做成高饱和色块。

按钮类型用途视觉建议注意事项
Primary页面主任务主色背景 + 白字同一操作区只放一个
Secondary次要但常用白底 + 边框不要抢主按钮
Tertiary/Ghost弱操作透明或文字按钮适合“更多、查看日志、展开”
Danger删除、撤销授权、清空红色或红色文字危险操作应远离主操作，并有确认
Icon Button工具栏、表格操作24/32/40px 点击区重要动作需要 tooltip 或文字

按钮必须覆盖的状态

Default保存
Hover保存
Focus保存
Loading处理中...
Disabled不可提交
Danger删除项目

09表单：减少用户犯错，而不是只摆输入框

表单是最容易体现专业度的组件。好的表单必须考虑 label、帮助文本、placeholder、校验、错误提示、禁用、只读、自动填充、提交后反馈。

坏例子：只靠 placeholder，错误不明确

密码
验证码
确定重置返回

问题：用户输入后提示消失；错误出现时不知道哪个字段错；按钮没有优先级。

好例子：字段结构完整
name@example.com
用于接收登录验证码。
••••
密码至少需要 8 位，并包含数字。
登录忘记密码

改进：label 永久可见；帮助文本和错误文本靠近字段；主按钮唯一。

字段部分该怎么做常见坑
Label字段名永远可见只用 placeholder 代替 label
Placeholder只放示例，不放关键说明输入后消失，用户忘记规则
Help Text解释格式、用途、限制把重要规则藏到提交后才提示
Error说明哪里错、为什么错、怎么改只写“输入错误”
Required统一说明必填或标记 *只用红色，没有文字说明
Submit提交中禁用 + loading + 防重复提交用户连续点，产生重复请求

10导航：帮助用户知道“我在哪、能去哪”

导航不是把链接放一排。它要回答三个问题：当前位置、可去位置、信息结构。后台产品通常需要侧边栏；内容站需要目录和面包屑；移动端需要底部导航或抽屉。

顶部导航

适合营销站、轻量 SaaS、少量一级页面。注意不要塞太多入口。

侧边导航

适合后台、开发者工具、多模块系统。需要分组、当前项、折叠策略。

面包屑

适合层级深的内容、管理系统详情页。让用户知道路径和返回方式。

导航 checklist：当前页面高亮了吗？一级/二级层级清楚吗？命名是用户语言还是技术语言？移动端怎么收起？权限不足的菜单是隐藏、禁用还是提示？

11表格、列表、卡片：数据界面的基本功

程序员经常做管理系统，表格尤其重要。表格不是把所有格子加边框，而是让用户快速扫描、比较和操作。

对象对齐样式注意事项
文本左对齐主文本深色，辅助文本浅一些长文本省略但提供查看方式
数字/金额右对齐使用千分位，必要时等宽数字单位统一，不要一会 ¥ 一会 元
状态居中或左对齐浅底色标签 + 深色文字不要用纯色大块刺激用户

表头与内容一致12/13px，500/600，浅灰底可排序字段要有明确图标

用户操作后必须知道结果。反馈的层级越高，对用户打断越强。不要把所有反馈都做成弹窗。

Progress耗时任务必须给进度或至少给 loading。

13状态设计：默认态之外才见产品功力

真实产品不是永远有数据、永远请求成功。你必须设计空、载入、错误、无权限、禁用、离线、部分成功、长数据、极端值。

∅
空状态

解释为什么空、下一步做什么。例：暂无订单，创建第一笔订单。

加载态

加载超过 300～500ms 就应该有反馈。

错误态

加载失败
网络可能不稳定，请稍后重试。

重新加载
无权限

说明缺少什么权限、向谁申请、是否可切换账号。

禁用态

禁用不等于沉默。最好说明为什么不能点。

部分成功

批量操作常见：成功 18 项，失败 2 项，提供失败详情。

14响应式与移动端：不是把桌面等比例缩小

响应式设计不是简单 width: 100%。移动端有手指、拇指热区、较短阅读宽度、较慢输入、较弱网络、系统键盘遮挡等问题。

桌面端

  • 可以多列布局
  • 表格密度可更高
  • 鼠标 hover 可辅助说明
  • 快捷键和批量操作更常见

移动端

  • 单列优先
  • 按钮高度 44/48px 更稳
  • 表格转卡片或横向滚动
  • 关键操作放在拇指容易触达区域

可访问性不是“照顾少数人”，而是让更多人在更多场景下能用：弱光、强光、手抖、色弱、键盘操作、屏幕阅读器、临时受伤、移动设备。

维度检查项实现建议
颜色文本对比度足够，不只靠颜色传达状态红色错误 + 图标 + 文案；图表加标签

16图标、图片、插画：统一风格，不要混搭

图标和图片最容易破坏界面一致性。常见问题是：线性图标和实心图标混用、不同粗细、不同尺寸、照片和 3D 插画混搭、图片裁切不一致。

图标规范

  • 统一图标库
  • 统一尺寸：16/20/24
  • 统一 stroke 宽度
  • 重要操作不要只有图标
  • 图标和文字 gap 6/8px

图片规范

  • 统一比例：1:1、4:3、16:9
  • 统一圆角
  • object-fit: cover
  • 头像、封面、商品图职责不同
  • 注意暗色/亮色图片上的文字可读性

17动效：反馈，不是表演

动效应该帮助用户理解状态变化，而不是展示技术。大多数 UI 动效都应该短、轻、可打断。

场景建议时长用途
Hover100～150ms提示可交互
展开/收起150～250ms表达空间变化
弹窗出现180～240ms引导注意力
页面切换200～350ms建立方向感
骨架屏循环但低干扰降低等待焦虑
不要：按钮点击后先播放 800ms 动画再执行；列表每一项都乱飞；删除成功后没有任何反馈；不尊重用户的减少动态偏好。

18数据可视化：图表也属于 UI

图表的目标不是“画得炫”，而是帮助比较、发现异常、解释趋势。颜色、坐标、标签、单位、排序都很重要。

坏例子：颜色太多，信息噪音大
好例子：用少量颜色突出重点
图表类型适合回答避免
折线图趋势如何变化？太多线、无图例、无单位
柱状图类别之间谁大谁小？3D 柱、随机颜色、排序混乱
饼图少量类别占比超过 5 类、差异很小时使用
表格精确读取数据把所有数据都强行图表化
热力图密集矩阵模式色阶无说明、色弱不可读

19文案与微文案：让用户知道发生了什么

很多 UI 丑，不是视觉丑，而是文字不清楚。按钮文案、错误提示、确认弹窗、空状态说明，都需要明确。

场景差文案好文案
错误Error订单加载失败，请检查网络后重试。
删除确认确定？删除这个项目？删除后无法恢复。
空状态暂无数据还没有订单。创建第一笔订单后，数据会显示在这里。
按钮提交保存修改 / 创建订单 / 发送邀请
禁用说明按钮灰掉请先填写邮箱地址，才能发送邀请。

20常见页面模式：不同页面有不同设计重点

登录页

核心是信任与低摩擦：清晰标题、少字段、错误靠近字段、找回密码、第三方登录、移动端键盘适配。

Dashboard

核心是概览：关键指标、趋势、异常、下一步行动。不要把所有图表平铺。

列表页

核心是查找与批量操作：筛选、排序、搜索、分页、状态、空结果、批量选择。

详情页

核心是上下文：主信息、关联信息、操作区、历史记录、危险操作分离。

设置页

核心是分组与安全：账户、通知、隐私、权限、账单；危险设置要确认。

编辑器/工作台

核心是效率：快捷键、侧栏、工具栏、焦点模式、撤销、自动保存、版本历史。

21风格不是随便选：产品气质如何影响视觉

同样的功能，可以有不同风格。但风格必须服务产品定位。B2B 后台通常需要稳定、可信、克制；消费类 App 可以更柔和、更情绪化；内容站可以更重视阅读节奏。

处理任务

消费型 / 社区

大圆角、柔和背景、更多情绪化插画、强调亲和。

开始体验

内容型 / 品牌

可使用衬线标题、更大留白、更强排版节奏。

23设计评审 checklist：每次提交前逐项检查

基础视觉 checklist
  • 页面第一眼能看出主标题和主操作吗？
  • 字号是否来自固定阶梯，而不是随机值？
  • 字重是否克制？是不是所有东西都加粗了？
  • 同组更近、异组更远了吗？
  • 元素是否有共同对齐线？
  • 一个区域是否只有一个主按钮？
  • 颜色是否都有角色？有没有随便出现的颜色？
  • 边框和阴影是否过重？
  • 圆角是否统一？
组件与状态 checklist
  • 按钮是否有 hover / active / focus / disabled / loading？
  • 输入框是否有 default / focus / error / disabled / readonly？
  • 表单错误是否靠近字段，并说明如何修正？
  • 表格是否有空数据、加载中、加载失败、长文本、批量操作？
  • 弹窗是否有明确标题、影响说明、主/次操作？
  • 危险操作是否二次确认或可撤销？
  • Toast 是否不会遮挡关键操作？
可访问性 checklist
  • 普通文本对比度是否满足至少 4.5:1？
  • 大文本是否至少 3:1？
  • 可点击目标是否至少 24×24 CSS px，移动端是否接近 44/48px？
  • 是否可以只用键盘完成主要流程？
  • 焦点状态是否明显？
  • 是否只靠颜色表达错误/成功/选中？
  • 图片是否有合理 alt？
  • 动效是否尊重 prefers-reduced-motion？
产品边界 checklist
  • 空状态是否告诉用户下一步？
  • 加载慢时是否有进度或骨架屏？
  • 失败后是否能重试？
  • 无权限时是否告诉用户如何申请权限？
  • 长名称、超大数字、多语言、窄屏是否会破版？
  • 弱网、重复点击、提交中刷新页面是否处理？

25训练路线：如何真正提高 UI 品味

“品味”不是空想出来的，是通过拆解、临摹、复盘建立模式库。程序员可以用逆向工程的方法练 UI。

第 1 阶段：临摹组件

按钮、输入框、卡片、表格、弹窗、Toast。目标是做到和成熟产品相似。

第 2 阶段：临摹页面

登录页、设置页、列表页、详情页、Dashboard。重点拆解间距、层级、状态。

第 3 阶段：建立系统

把临摹出的规律变成 token 和组件库，之后所有页面都复用。

练习做法检查重点
截图测量找一个好页面，测字号、间距、按钮高度、卡片 padding不是评价好看，而是提取规则
黑白灰重做不用主色，只用文字、线、留白做页面没有颜色还能看出层级吗？
一页三密度同一页面做紧凑、标准、宽松三个版本理解密度对体验的影响
状态补全给一个表格补齐 empty/loading/error/permission从 Demo 变成真实产品
组件抽象把页面里的按钮、输入、卡片抽成统一组件避免样式分叉

26复杂组件与高级交互：越复杂越要克制

当产品变复杂，UI 问题通常不再是“按钮好不好看”，而是用户能不能在复杂流程里保持方向感。复杂组件要优先解决：范围、反馈、撤销、错误恢复、可发现性。

组件/交互关键设计点常见坑
Dropdown / Select选项数量少时使用；支持搜索时明确输入行为；选中项可见几十上百个选项还用普通下拉
Combobox输入 + 选择混合；需要键盘上下选择、回车确认、Esc 关闭看起来像输入框，实际行为像选择器，用户困惑
Tabs同级内容切换，不改变大任务；当前项明显把流程步骤做成 tabs，用户误以为可随意跳转
Accordion用于渐进披露；标题要能概括隐藏内容把关键内容默认折叠，导致用户找不到
Drawer适合编辑详情、筛选、上下文面板和 Modal 混用，没有关闭/保存规则
Wizard / Stepper适合线性流程；显示进度、可返回、可保存草稿步骤太多且无法中途退出
Command Palette适合高级用户；支持搜索命令、快捷键、最近使用把基础功能只藏在命令面板里
Drag & Drop提供拖动手柄、目标区域、释放反馈、键盘替代方案只能拖，不能用按钮移动，触屏和键盘不可用
File Upload文件类型、大小、进度、失败原因、重试、取消上传失败只显示“失败”
复杂组件原则：越复杂，越不能只设计默认态。必须考虑 keyboard、loading、empty、error、overflow、mobile、权限、撤销、确认。

27搜索、筛选、排序、分页：列表页的灵魂

列表页最重要的不是表格本身，而是用户如何缩小范围、找到目标、批量处理。搜索、筛选、排序、分页如果设计不好，数据越多越难用。

功能设计规则工程注意
搜索说明搜索范围；支持清空；无结果要给建议防抖、请求取消、历史关键词、输入法合成事件
筛选筛选条件可见；已选条件可移除；支持一键清空URL query 同步，便于分享和刷新恢复
排序当前排序字段和方向明显；默认排序符合用户任务后端排序字段与 UI 文案解耦
分页总数、当前页、每页数量、跳转明确注意删除最后一条后的页码回退
无限滚动适合信息流，不适合强任务表格返回位置恢复、加载失败重试、页脚不可达问题
批量选择选中数量可见；全选当前页/全选全部要区分跨页选择状态、权限差异、部分失败结果
搜索订单号 / 客户名状态：已支付 ×时间：近 7 天 ×清空筛选导出结果

好的筛选区要让用户知道：当前查了什么、还可以改什么、如何撤销筛选。

暗色模式不是简单把白变黑、黑变白。你要重新定义背景层级、文字强度、边框、阴影、状态色和图表色。深色背景上的高饱和颜色会显得刺眼。

29国际化、本地化、数字与时间：细节决定可信度

很多工程师忽略格式问题，但格式直接影响专业感。日期、时间、金额、单位、姓名、地址、多语言长度都会改变 UI。

类型设计注意例子
日期明确时区；相对时间和绝对时间不要混淆2026-07-06 17:00 JST / 3 分钟前

30性能感知：UI 要管理等待感

真实速度重要，感知速度也重要。用户不只关心是否加载完成，还关心系统是否“还活着”、是否可取消、是否知道进度。

等待类型推荐反馈注意事项
瞬时操作按钮 active/hover 即可不要过度打扰
短请求按钮 loading、局部 spinner防止重复提交
列表加载骨架屏或保留旧数据 + 更新提示避免整页白屏
长任务进度条、阶段说明、可取消、完成通知后台任务要允许离开页面
失败说明原因、影响、重试方式不要只写“系统错误”
工程实践：优先做局部加载，少做全屏 loading；保留用户已输入内容；失败后提供重试；耗时任务给进度或阶段文案。

31设计系统治理：防止 UI 债务越滚越大

当项目变大，UI 会像代码一样产生债务：重复组件、颜色分叉、样式覆盖、例外越来越多。治理的核心是：规则可查、组件可复用、变更可追踪。

治理对象要维护什么程序员做法
Token颜色、字号、间距、圆角、阴影集中在 CSS variables / theme.ts
组件Button、Input、Table、Modal、Toast 等Storybook / 文档页 / 示例状态
模式登录、搜索筛选、删除确认、批量操作写成 pattern 文档，不只写组件文档
版本破坏性变更、废弃样式、迁移说明Changelog + codemod 或 lint 规则
质量可访问性、视觉回归、响应式axe、Playwright screenshot、lint token

32设计与开发协作：从视觉稿到真实产品

真实项目里，UI 不会只停留在设计稿。你要关心设计如何被实现、如何适配数据、如何覆盖边界状态、如何在版本迭代中保持一致。

开发接设计稿时要问

  • 组件是否已有？没有是否要新增？
  • 所有状态是否齐全？
  • 长文本、空数据、错误态如何展示？
  • 移动端/窄屏如何适配？
  • 权限差异如何处理？
  • 哪些样式是 token，哪些是一次性？

提交 UI PR 时要附

  • 截图：桌面 + 移动端
  • 状态：default/loading/error/empty
  • 交互：hover/focus/disabled
  • 可访问性：键盘与对比度
  • 数据：长文本、大数字、无数据
  • 说明：是否新增 token/组件
最终目标：让 UI 像代码一样可 review。不要用“我觉得好看/不好看”讨论，而用“层级、对齐、间距、对比、状态、可访问性、任务效率”讨论。

35最后的最短原则

当你觉得页面丑，不要马上换颜色。按这个顺序修：内容 → 层级 → 对齐 → 间距 → 字号 → 字重 → 颜色 → 状态 → 可访问性 → 风格。前八项没做好，任何“高级配色”都会变成装饰噪音。
