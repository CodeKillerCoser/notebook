01 — String 处理与字符遍历

Rust 的 String 是 UTF-8 编码的字节序列。遍历它时有两种截然不同的视角： 逐字符（char，Unicode 标量值）和逐字节（u8）。选择哪种方式 直接决定正确性和性能。

1. 三种遍历方式的对比

1.1 char_indices() — 推荐的首选

let s = String::from("a b c");
for (idx, ch) in s.char_indices() {
    // idx: usize  — 字符的字节偏移
    // ch:  char   — 经 UTF-8 解码的 Unicode 标量值
    println!("s[{}] = {}", idx, ch);
}
// 输出:
// s[0] = a
// s[1] =
// s[2] = b
// s[3] =
// s[4] = c
✅ 正确：每个 ch 都是一个完整的 Unicode 字符，空格、中文、emoji 都会被正确识别。内部做了 UTF-8 解码，有适度开销，但对 LeetCode 级数据量完全可以忽略。

1.2 as_bytes() — 速度最快，但牺牲通用性

let s = String::from("abcabc");
let bytes: &[u8] = s.as_bytes();
for (idx, &ch) in bytes.iter().enumerate() {
    // idx: usize  — 字节偏移
    // ch:  u8     — 原始字节，未解码
}
⚡ 快的原因：不做 UTF-8 解码，iter() 直接走指针偏移， 每个 u8 就是一个 mov 指令。

❌ 局限：多字节字符会被错误拆分。例如 'é' 编码为 [0xC3, 0xA9]，在字节视角下被当成两个独立的字节，逻辑错误。

字符UnicodeUTF-8 字节as_bytes 视角
'a'U+0061[0x61]1 个 u8 = 0x61 ✅
' ' (空格)U+0020[0x20]1 个 u8 = 0x20 ✅
'é'U+00E9[0xC3, 0xA9]2 个 u8 ❌ 被拆分
'你'U+4F60[0xE4, 0xBD, 0xA0]3 个 u8 ❌ 被拆分

1.3 chars().nth(i) — O(n²) 陷阱

// ❌ 错误：nth() 每次从头迭代，总复杂度 O(n²)
for i in 0..s.len() {
    if let Some(ch) = s.chars().nth(i) {
        // 仅在特殊场景可用，循环中坚决避免
    }
}

String::chars() 返回的是一个从零开始的 UTF-8 迭代器。 调用 nth(i) 会从头跳过前 i 个字符——这是一次 O(n) 操作。 放在 for i in 0..len 循环中，总复杂度退化为 O(n²)。

2. 模式解构中的 &ch

字节数组版的一个关键细节：

let s: &[u8] = "abc".as_bytes();

// s.iter() 返回 Iterator<Item = &u8>
// .enumerate() 返回 Iterator<Item = (usize, &u8)>

for (right, &ch) in s.iter().enumerate() {
//             ^^
// 这行模式匹配干了什么？
//   1. s.iter() 每轮产生 &u8
//   2. enumerate() 包装成 (usize, &u8)
//   3. &ch 模式 —— 剥掉 &，让 ch: u8
//   4. 之后可以直接 ch as usize
}
核心规则：s.iter() 让你进入「引用的世界」， 模式里的 & 把你拉回来。不写 & 的话， ch 就是 &u8，不能直接 as usize。
