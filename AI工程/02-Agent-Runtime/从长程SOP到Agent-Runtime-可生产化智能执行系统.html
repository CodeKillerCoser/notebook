从长程 SOP 到 Agent Runtime：如何设计一个可生产化的智能执行系统

很多团队一开始会把 Agent 失败归因于“模型不够强”或“prompt 不够好”。但一旦任务变成长程 SOP，核心问题就从 prompt engineering 变成 runtime engineering。

以“基于用户输入抓取日志、清洗数据、生成报告、展示问题、做需求分析”为例，报告质量不稳定通常表现为三类问题：报表或图表丢失，分析停留在表层，SOP 步骤不完整。表面上看，这是 AI 没有“认真执行”；本质上看，是系统缺少一个能管理长程智能任务的运行时。

Agent Runtime 的本质不是让模型“多想几步”，而是让模型的每一次行动都在真实系统中被验证、授权、记录、评估、提交或回滚。

因此，一个生产级 Agent 系统的关键原则应当是：模型只提议，runtime 才提交。模型可以提出下一步行动，例如查日志、生成图表、请求更多数据、重写报告段落；但是否执行、如何执行、执行后是否进入下一步，应该由 runtime 负责。

01 · Definition

1什么是 Agent Runtime？

Agent Runtime 是一个持续执行环境。它接收目标、约束、状态和工具集合，驱动一个或多个 Agent 在循环中行动，并通过验证、权限、状态提交、重试、回退和人工介入来保证任务可靠推进。

可以把它理解为三类系统能力的组合：

01

LLM Agent SDK

提供模型、工具、handoff、guardrail、structured output 等智能应用抽象。

02

Workflow Engine

提供状态机、DAG、重试、检查点、持久化、失败恢复和长任务调度。

03

LLMOps / Governance

提供 trace、eval、成本、权限、审计、安全、版本和线上质量监控。

业界的方向也在收敛。OpenAI 将 Agents 描述为能够规划、调用工具、跨专家协作并保留足够状态以完成多步骤工作的应用；OpenAI Agents SDK 也把 agent、tools、handoffs、guardrails、sessions、human-in-the-loop 和 tracing 作为核心能力。OpenAI Agents 指南、OpenAI Agents SDK 文档

LangGraph 将自己定位为面向长程、有状态 Agent 的编排 runtime，强调 durable execution、human-in-the-loop 和 persistence；Temporal 则代表传统 durable execution 路线，强调在间歇性失败中可靠执行 workflow。LangGraph Overview、Temporal

关键判断：如果你的任务只需要一次模型调用和少量应用侧逻辑，runtime 可以很薄；如果你的应用要管理编排、工具执行、审批、状态、产物和失败恢复，就需要认真设计 Agent Runtime。
02 · Core Loop

2标准执行闭环：State → Action → Observation → Evaluation → Commit

一个可靠的 Agent Runtime 不是简单的 model → tool → model 循环，而是一个带裁判和状态提交机制的闭环。

这个闭环可以用伪代码表示：

while not run.done:
    state = state_store.load(run_id)
    context = context_builder.build(state)

    proposal = agent.act(context)                # 模型只提出候选行动
    policy = policy_engine.check(proposal, state)

    if policy.decision == "deny":
        state = router.handle_denial(state, policy)
        continue

    if policy.decision == "requires_approval":
        human_loop.pause(run_id, policy)
        break

    observation = executor.execute(proposal)     # runtime 执行真实动作
    evaluation = evaluator.evaluate(state, proposal, observation)
    route = router.decide(evaluation)

    state = reducer.apply(state, proposal, observation, evaluation, route)

    if route.commit_artifacts:
        artifact_store.commit(observation.artifacts)

    state_store.save_checkpoint(run_id, state)
    event_log.append(state, proposal, observation, evaluation, route)
关键差异：普通 Agent Loop 关注“模型下一步想做什么”；Agent Runtime 关注“这个行动是否允许执行、执行后是否可信、失败时如何处理、哪些产物可以提交”。
03 · Architecture

3总体架构：Control Plane + Data Plane

如果把 Agent Runtime 做成平台，建议分成两层：Control Plane 管配置、版本和治理；Data Plane 负责运行、执行和观测。

04 · Runtime Modules

4Agent Runtime 的十二个核心模块

一个严肃的 Agent Runtime 至少需要以下模块。它们不一定一开始都要完整实现，但架构设计时要给它们预留位置。

模块职责为什么重要
Run Manager创建、暂停、恢复、取消、终止一次任务实例。把 Agent 执行从普通函数调用升级为可管理的 run。
State Store保存目标、当前节点、变量、质量门禁、审批状态、预算。上下文不是事实源，数据库状态才是事实源。
Event Log记录模型调用、工具调用、校验、路由、人工审批、产物提交。支持审计、回放、debug、eval 和恢复。
Context Builder从状态、产物、记忆、工具和策略构造本轮上下文。控制模型看到什么，避免上下文污染。
Model Gateway统一模型调用、fallback、token 统计、prompt 版本、结构化输出解析。避免模型调用散落在业务代码里。
Tool Gateway统一工具注册、参数校验、执行隔离、重试、超时、脱敏。工具调用是 Agent Runtime 最大风险点之一。
Policy Engine处理权限、风险、预算、数据边界、人工审批。模型不能自己批准自己的行动。
Artifact Store保存中间产物、版本、校验状态和血缘。防止报告、图表、数据、计划丢失。
Evaluator做 schema、数据、产物、语义、业务、安全和成本评估。闭环需要裁判，而不只是模型自我反思。
Router / Scheduler决定 continue、retry、repair、replan、rollback、human、terminate。不同失败需要不同处理策略。
Human Manager把人工审批、修改状态、修改产物做成 pause/resume。人工介入不是聊天，而是状态操作。
Observability / Replay追踪、可视化、成本、延迟、失败归因、回放。没有 trace，就无法定位为什么失败。

OpenAI Agents SDK 的 tracing 会记录 agent run 中的 LLM generations、tool calls、handoffs、guardrails 和 custom events；这说明 trace 在生产 Agent 中不是锦上添花，而是基础设施。OpenAI Agents SDK Tracing

06 · Tools & Policy

6工具调用：模型可以提出，但不能直接执行

Agent Runtime 的高风险点通常不是模型写错几句话，而是模型调用了真实工具：查数据库、发请求、读文件、写系统、触达用户、修改状态。Runtime 必须用 Tool Gateway 和 Policy Engine 把这些动作托管起来。

MCP 规范把 tools 描述为让模型与外部系统交互的能力，例如查询数据库、调用 API 或执行计算；每个工具由名称唯一标识，并包含描述 schema 的元数据。MCP Tools Specification 但协议层的工具 schema 只是起点，生产 runtime 还需要处理权限、风险、审计、幂等、脱敏和审批。

{
  "tool_id": "query_logs",
  "kind": "function",
  "input_schema": "QueryLogsInput",
  "output_schema": "QueryLogsOutput",
  "side_effect": "read_only",
  "risk_level": "medium",
  "required_scopes": ["logs:read"],
  "timeout_seconds": 60,
  "retry_policy": {
    "max_retries": 2,
    "backoff": "exponential"
  },
  "idempotency": {
    "required": true,
    "key_fields": ["query_hash", "time_range"]
  },
  "output_policy": {
    "redact_pii": true,
    "mark_untrusted_content": true
  }
}
检查项问题Runtime 动作
Schema参数是否合法？字段是否完整？拒绝或要求模型 repair。
Permission用户和 Agent 是否有权限调用？deny 或请求审批。
Risk是否有副作用？是否触达外部系统？按风险分级执行。
Budget是否超过成本、token、时间或工具调用次数？降级、终止或请求确认。
Data Boundary是否越权访问敏感字段或跨租户数据？拒绝、脱敏或缩小范围。
Idempotency重试是否会造成重复副作用？要求幂等 key 或禁止自动重试。
运行时原则：Agent 可以解释“为什么要调用工具”，但不能自己决定“是否允许调用工具”。这属于 runtime 的权限和治理职责。
07 · Evaluation

7Evaluator：闭环的裁判，而不是最后的打分器

很多团队把 eval 当成上线后的人工抽检或质量报表。对 Agent Runtime 来说，eval 应该是执行过程中的控制信号。

Structured Outputs 能让模型输出符合开发者定义的 JSON Schema，减少遗漏 required key 或产生非法枚举值的问题。OpenAI Structured Outputs 但 schema 只能保证结构，不能保证业务质量。因此 runtime 还需要更多 evaluator。

确定性 Evaluator

  • JSON Schema 是否通过
  • 文件是否存在
  • 图表是否被报告引用
  • 时间范围是否覆盖需求
  • 工具调用是否越权

语义 / 业务 Evaluator

  • 结论是否有证据
  • 是否包含基线和分群
  • 是否区分事实与假设
  • 建议是否可执行
  • 是否说明置信度和限制

Evaluator 输出不能只是“好 / 不好”，而要给 Router 可执行的 repair plan。

{
  "status": "failed",
  "score": 0.72,
  "issues": [
    {
      "issue_id": "INSIGHT_QUALITY_003",
      "severity": "high",
      "type": "missing_baseline",
      "message": "洞察 3 缺少历史基线对比",
      "repair_action": {
        "type": "query_more_data",
        "target": "historical_baseline"
      }
    },
    {
      "issue_id": "REPORT_004",
      "severity": "medium",
      "type": "chart_not_referenced",
      "message": "错误码分布图已生成，但未在根因分析章节中引用",
      "repair_action": {
        "type": "revise_report_section",
        "target_section": "根因分析"
      }
    }
  ],
  "next_route": "repair"
}
关键点：Evaluator 不是审稿意见箱，而是 runtime 的路由输入。它决定任务是进入下一步、局部修复、重新规划、回滚、降级输出还是请求人工。
08 · Graph + Loop

8不要在 Workflow 和 Agent Loop 之间二选一

一个好的 Agent Runtime 应该同时支持 Graph Execution 和 Loop Execution。

执行形态适合场景优点风险
Graph Execution路径清晰、阶段明确的 SOP。稳定、可审计、可恢复。对开放任务不够灵活。
Loop Execution探索式分析、根因假设、工具选择。灵活、能动态补查数据。容易无限循环或漂移。
Graph + Loop生产级长程智能任务。外层稳定，局部灵活。需要 runtime 设计更成熟。

Anthropic 在 agentic systems 的实践中也区分 workflows 和 agents：workflows 通过预定义代码路径编排 LLM 与工具；agents 则由 LLM 动态决定过程和工具使用。生产系统通常不应盲目追求完全自主，而应把明确路径交给 workflow，把不确定判断交给局部 agent loop。Anthropic: Building Effective Agents

09 · Multi-agent

9多 Agent 协作：不要自由聊天，要通过 Artifact 协作

多 Agent 最常见的误区是把多个角色放进一个群聊里自由讨论。这种模式演示时很炫，但生产中难以审计、难以复现，也难以控制成本和责任边界。

更稳的做法是：Agent 之间不直接共享无结构聊天记录，而是通过 runtime 管理的 artifact、state 和 handoff 协作。

{
  "type": "Handoff",
  "from_agent": "report_writer",
  "to_agent": "report_critic",
  "reason": "draft_report_ready_for_review",
  "input_artifacts": [
    "report_draft.v2.md",
    "report_manifest.v1.json",
    "chart_manifest.v1.json"
  ],
  "allowed_actions": ["review", "request_revision"],
  "output_schema": "ReviewResult"
}
10 · Failure Model

10失败不是异常，而是 Runtime 的一等公民

长程 Agent 必然会失败。成熟 runtime 的差别不在于“永不失败”，而在于它是否能识别失败类型，并采取正确动作。

失败类型例子正确动作
临时失败查询超时、API 429、网络错误。retry
格式失败JSON 不合法、字段缺失、枚举值非法。repair
语义失败结论无证据、建议不对应问题。deepen / regenerate
数据失败样本量过低、字段缺失、时间范围不覆盖。replan
产物失败图表文件缺失、报告未引用图表。local repair
权限失败越权访问数据、查询敏感字段。deny / approval
预算失败超 token、超成本、循环无进展。degrade / terminate
11 · Memory & Security

11记忆与安全：Context 不是垃圾桶，Memory 不能自动污染

Agent Runtime 需要区分四类 memory：

Session Memory

用户对话历史，适合延续交流，但不足以表达任务状态。

Run Memory

当前 run 的目标、计划、失败记录、待修复项和确认假设。

Domain Memory

业务规则、指标口径、日志字段定义、常见异常模式。

Episodic Memory

历史任务经验，例如某类任务经常漏查某个字段或漏生成某类图表。

Context Builder 的职责，是把这些材料按当前节点需要进行裁剪、降噪和权限过滤。Runtime 不能把所有历史消息、所有日志、所有工具输出、所有记忆全部塞给模型。

{
  "task": "基于已提交的指标数据生成候选洞察",
  "current_stage": "generate_insights",
  "allowed_inputs": [
    "metric_summary.v1.json",
    "data_quality_report.v1.json"
  ],
  "forbidden_inputs": ["raw_sensitive_logs"],
  "allowed_tools": ["query_segment_metrics", "get_release_timeline"],
  "output_schema": "InsightCards",
  "quality_rules": [
    "每个 insight 必须有 baseline",
    "每个 insight 必须有 evidence",
    "candidate insight 不能直接写入 final report"
  ],
  "budget": {
    "max_tool_calls": 5,
    "max_tokens": 12000
  }
}

安全模型

安全不能只靠 prompt。OWASP 将 prompt injection 定义为用户提示改变 LLM 行为或输出的风险，并指出这些输入即使对人类不可见，只要被模型解析也可能影响模型。OWASP LLM01: Prompt Injection

  • 工具采用 capability-based access，不同 agent 和用户只能访问被授权工具。
  • 工具参数必须 schema 校验，高风险工具必须人工审批。
  • 工具输出必须带来源和可信级别，外部内容永远不能升级为系统指令。
  • 敏感数据在进入模型前脱敏，secrets 永远不进入模型上下文。
  • 所有工具调用、模型输出、权限决策和人工审批都进入审计日志。
12 · Runtime IR

12一个推荐的 Agent Runtime IR

如果你要把 Agent Runtime 做成平台，建议设计一个中间表示：Agent Runtime IR。它把 agent、workflow、tool、validator、router、artifact 统一描述，方便版本化、静态检查、可视化、回放和部署。

runtime_program:
  id: intelligent_report_workflow
  version: v1

  inputs:
    - user_request
    - user_context

  state_schema: ReportRunState

  nodes:
    - id: parse_requirement
      type: llm
      agent: requirement_parser
      output: requirement_spec
      validators:
        - requirement_schema
        - scope_policy
      routes:
        pass: build_query_plan
        fail: repair_requirement

    - id: build_query_plan
      type: llm
      agent: query_planner
      output: query_plan
      validators:
        - query_plan_schema
        - query_policy
      routes:
        pass: execute_query
        fail: repair_query_plan

    - id: execute_query
      type: tool
      tool: query_logs
      output: raw_data
      validators:
        - raw_data_coverage
        - raw_data_schema
      routes:
        pass: clean_data
        fail: replan_query

    - id: clean_data
      type: code
      output: cleaned_data
      validators:
        - cleaning_diff_check
        - data_quality_check
      routes:
        pass: generate_insights
        fail: repair_cleaning

    - id: generate_insights
      type: loop
      loop:
        max_iterations: 5
        body:
          - propose_claim
          - gather_evidence
          - challenge_claim
          - evaluate_claim
      output: accepted_insights
      validators:
        - insight_schema
        - evidence_completeness
        - business_impact_eval
      routes:
        pass: plan_report
        fail: query_more_data

    - id: plan_report
      type: llm
      agent: report_planner
      output: report_manifest
      validators:
        - required_sections_check
        - required_charts_check
      routes:
        pass: generate_charts
        fail: repair_report_manifest

    - id: generate_charts
      type: code
      output: chart_manifest
      validators:
        - chart_file_existence
        - chart_data_lineage
      routes:
        pass: write_report
        fail: regenerate_missing_charts

    - id: write_report
      type: llm
      agent: report_writer
      output: report_draft
      validators:
        - report_manifest_check
        - chart_reference_check
        - source_traceability_check
      routes:
        pass: review_report
        fail: repair_report

    - id: review_report
      type: llm
      agent: report_critic
      output: review_result
      validators:
        - review_schema
      routes:
        pass: publish
        fail: revise_report
IR 的价值：它让 Agent 应用从“几段 prompt + 几个工具函数”升级为可以被解释执行、审计、测试、版本化和治理的 runtime program。
13 · Maturity Model

13Agent Runtime 成熟度模型

不需要一开始就做大而全的平台。可以按成熟度逐步建设。

级别能力适合阶段
Level 1：Agent RunnerAgent、tools、structured output、basic tracing。Demo、内部工具、短任务。
Level 2：Stateful RuntimeRunState、ArtifactStore、EventLog、Evaluator、Router。真实业务长任务。
Level 3：Durable Runtimecheckpoint、pause/resume、crash recovery、replay、局部修复。生产级长程任务。
Level 4：Enterprise Runtime多租户、RBAC/ABAC、policy engine、audit、eval platform、成本治理。组织级 Agent 平台。
14 · Implementation

14MVP：先做最小内核，而不是大而全平台

如果要落地，不建议一开始就做“完整 Agent OS”。最小可用 runtime 先做好六件事：

1. RunState

任务目标、当前阶段、变量、门禁、预算、待修复项。

2. EventLog

所有模型调用、工具调用、校验、路由都事件化。

3. ArtifactStore

所有中间结果都落盘、版本化、可校验。

4. ToolGateway

统一参数校验、权限校验、执行、重试和输出脱敏。

5. Evaluator

做 schema、产物、数据、语义和业务质量校验。

6. Router

把失败路由到 retry、repair、replan、rollback 或 human。

一个可执行的 MVP 流程

create_run
  ↓
load_state
  ↓
build_context
  ↓
call_agent
  ↓
validate_action
  ↓
execute_tool_or_node
  ↓
evaluate_result
  ↓
commit_artifact
  ↓
update_state
  ↓
emit_event
  ↓
route_next

当这六个组件跑通后，再逐步加入 memory、multi-agent handoff、human approval、parallel execution、durable recovery、eval platform 和 policy engine。

15 · Conclusion

15结语：Agent Runtime 是智能应用的操作系统雏形

当 AI 应用停留在一次性生成阶段，prompt、模型和工具函数就足够了。但当它开始执行长程 SOP、调用真实系统、处理产物、接受反馈、修复失败并交付可审计结果时，就需要 runtime。

Agent Runtime = Workflow Engine + Model Gateway + Tool Gateway + State Machine + Artifact System + Evaluator + Policy Engine + Observability。

最终的设计原则可以浓缩为十句话：

  1. 模型只提议，runtime 才提交。
  2. 状态不放在 prompt 里，状态放在 State Store。
  3. 上下文是动态构造的视图，不是事实源。
  4. 工具必须经过统一 Gateway 和 Policy。
  5. 每个中间产物都 artifact 化和版本化。
  6. 每一步都必须有 evaluator。
  7. 失败要分类处理：retry、repair、replan、rollback、human。
  8. 多 Agent 通过 artifact 和 handoff 协作，不靠自由群聊。
  9. Trace 和 Event Log 是基础设施，不是调试附属品。
  10. Eval 结果不仅用于评分，还用于控制 runtime 路由。

这就是从“会调用 LLM 的应用”走向“可生产化 Agent 系统”的关键跃迁。
