1一、总览：Agent 沙箱到底是什么

在 agent 系统里，sandbox 保护的不是模型内部的推理过程，而是模型发起的外部动作。模型可以提出“我要运行命令”“我要修改文件”“我要访问网站”“我要调用工具”，但这些动作不应该直接落到宿主机上，而应该经过 runtime、policy 和 sandbox 的组合。

Agent Sandbox = 隔离执行环境
              + 文件系统策略
              + 网络出口治理
              + 资源限制
              + 凭据代理
              + 审批流程
              + 审计与恢复

一个成熟的 agent runtime 大致可以拆成以下链路：

用户任务
  ↓
Agent Harness / Orchestrator
  - 调模型
  - 管上下文
  - 选择工具
  - 执行策略和审批
  - 记录 trace
  ↓
Policy / Gateway
  - 文件策略
  - 网络策略
  - 命令策略
  - 凭据代理
  - 人工确认
  ↓
Sandbox Session
  - 文件系统
  - shell / Python / Node
  - 浏览器 / GUI / 测试 runner
  - 资源限制
  - 临时状态
  ↓
Artifacts / Logs / Diff
  - stdout / stderr
  - 文件 diff
  - 生成产物
  - 审计记录

核心原则：不要相信模型一定不会做错事，而是让模型即使做错，影响范围也被限制在可控边界内。

2二、为什么 agent 比普通代码更需要沙箱

传统服务的执行路径通常是开发者提前写好、测试过、review 后部署。Agent 的不同点在于：它可能在运行时根据网页、issue、README、日志、用户文件和工具返回结果生成命令、脚本、补丁、浏览器动作或 MCP 工具调用。执行路径本身是动态的。

风险例子主要防线
误操作模型误删文件、改错目录、覆盖配置workspace-only、只读根文件系统、diff review
提示注入网页或项目文档诱导 agent 执行不该执行的动作默认禁网、文件隔离、输出审查、工具策略
供应链风险安装依赖时执行不可信脚本setup / agent 阶段分离、包源 allowlist、无长期凭据
主机逃逸恶意二进制攻击内核或容器 runtimeseccomp、LSM、gVisor、Kata、Firecracker、VM
敏感数据泄露读取本机配置、项目私密文件或长期凭据clearenv、禁止挂载宿主用户目录、credential broker
网络外传把项目内容或凭据发送到外部站点network namespace、egress proxy、domain allowlist
资源滥用无限循环、爆内存、创建大量子进程、写满磁盘cgroup、Job Object、timeout、quota
现实副作用提交代码、发消息、部署、调用云 APItool broker、最小 scope、human-in-the-loop

Prompt 是软约束，sandbox 是硬约束。“请不要读取敏感文件”不应该是主要防线；更好的设计是让沙箱里根本看不到敏感文件，让 agent 默认不能联网，让真实凭据留在可信 broker 里。

3三、成熟 agent sandbox 的运行时架构

一个成熟的 agent sandbox 不应该只是 subprocess.run() 外面套一层容器，而应该拆成控制面、执行面、策略面和产物面。

层职责不应该放什么
Control Plane模型调用、工具路由、任务状态、审批、trace、恢复不要和不可信代码放在同一个边界里
Execution Planeshell、Python、Node、浏览器、测试、构建、文件系统不要放长期凭据和宿主用户目录
Policy Plane文件、网络、命令、凭据、资源、工具权限不要只靠正则黑名单
Artifact Planestdout/stderr、diff、报告、生成文件、日志、快照不要让沙箱任意写回真实系统

从工程接口看，关键抽象不是“执行一段 shell 字符串”，而是一个明确的 session API：

SandboxSession:
  exec(argv, cwd, env, timeout, network_policy) -> ExecResult
  read_file(path) -> bytes
  write_file(path, bytes) -> void
  list_files(path) -> list
  diff() -> patch
  export_artifact(path) -> artifact
  destroy() -> void

这样你的 agent harness 不关心底层是 bubblewrap、Docker、gVisor、Kata 还是 Firecracker。先实现一个简单 runner，后续替换底层隔离后端即可。

4四、要管的不是 shell，而是能力

很多人第一次做 agent sandbox 时，会把注意力集中在“禁止哪些命令”。这只是辅助。模型可以通过 Python、Node、构建脚本、Git hook、依赖安装脚本或二进制工具绕过字符串黑名单。更正确的模型是 capability-based security：显式建模 agent 具有什么能力。

filesystem:
  read:
    - /workspace
  write:
    - /workspace
    - /tmp
  deny_host_user_data: true
  deny_runtime_sockets: true

network:
  default: deny
  allow_domains: []
  block_private_cidrs: true
  block_metadata_service: true

process:
  user: nonroot
  timeout_seconds: 600
  memory: 4GiB
  cpu: 2
  pids: 256

secrets:
  raw_values_inside_sandbox: false
  injection: brokered_proxy
  ttl: short

approvals:
  require_for:
    - new_network_domain
    - write_outside_workspace
    - production_api_call
    - destructive_command

在这个模型里，命令策略是前置判断；真正的硬边界来自 OS/container/VM、文件系统、网络、资源限制和凭据代理。

5五、三平台实现哲学对比

Linux、Windows、macOS 都能做沙箱，但底层哲学不同。

平台核心哲学关键机制最适合的 agent 形态
Linux给进程造一个新的小世界namespace、cgroup、capabilities、seccomp、AppArmor/SELinux/Landlock、container runtime、microVM云端 agent runner、代码执行器、CI-like coding agent、多租户 sandbox
Windows给进程一个受限身份，让它无法访问未授权对象Access Token、SID、DACL、Integrity Level、Restricted Token、AppContainer/LPAC、Job Object、Windows Container、Hyper-V、Windows SandboxWindows GUI agent、Win32 工具自动化、企业桌面自动化、Windows 容器任务
macOS签名应用声明少量 entitlement，其余默认被系统拒绝App Sandbox、entitlements、security-scoped bookmarks、TCC、SIP、Hardened Runtime、Seatbelt/sandbox-exec、XPC本地 Mac agent、IDE agent、保护用户文件和隐私资源

一句话：Linux 是“看不到”；Windows 是“看到了也打不开”；macOS 是“没 entitlement 和用户授权就不给”。

6六、Linux 沙箱机制详解

Linux 沙箱不是一个单独 API，而是一组内核原语的组合。核心包括 namespace、cgroup、capabilities、no_new_privs、seccomp 和 LSM。

6.1 Mount namespace：文件系统视图

Mount namespace 决定沙箱里能看到哪些目录、哪些目录只读、哪些目录可写。Agent sandbox 典型文件系统视图如下：

sandbox root
├── workspace/      rw bind mount，agent 的工作区
├── tmp/            tmpfs，任务结束销毁
├── usr/            ro bind mount，运行时程序
├── bin/            ro bind mount
├── lib/            ro bind mount
├── proc/           沙箱自己的 proc
└── dev/            最小设备节点

不要挂载宿主用户目录、浏览器 profile、运行时控制 socket 或宿主系统的可写路径。文件隔离不能只靠应用层 path check，因为 path check 可能被符号链接、路径跳转、挂载和竞态绕过；mount namespace 是内核层的文件系统视图隔离。

6.2 PID namespace：进程树隔离

PID namespace 让沙箱里的进程看到自己的 PID 空间。沙箱内 ps 看不到宿主机进程，沙箱内的进程管理动作也不会直接作用到宿主进程树。但 PID namespace 不防资源滥用，所以还要配合 pids.max 或进程数限制。

6.3 Network namespace：网络栈隔离

Network namespace 隔离网卡、路由表、防火墙规则、端口空间等。Agent 默认应该无网络。需要安装依赖时，可以把流程拆成 setup phase 和 agent phase：setup 阶段只允许有限包源；agent 自主执行阶段重新回到 no network。

默认：
  no network

需要安装依赖时：
  setup phase 允许受控包源
  agent phase 重新回到 no network

需要访问外部 API 时：
  sandbox -> egress proxy -> allowlist / audit / token broker -> internet

6.4 User namespace 与 capabilities

User namespace 允许沙箱内的 root 映射成宿主机上的普通用户，这是 rootless container 的基础。Capabilities 则把传统 root 权限拆成多个细粒度能力。Agent sandbox 默认应该 drop 所有 capabilities，尤其不要给 mount、网络管理、进程调试、内核模块、绕过文件权限等高风险能力。

6.5 cgroup v2：资源限制

Namespace 解决“看到什么”，cgroup 解决“能用多少”。Agent runner 至少要限制内存、CPU、进程数、IO、运行时间和输出大小。

资源常见控制项agent 风险
内存memory.max、memory.swap.max爆内存拖垮宿主
CPUcpu.max无限循环或构建任务吃满 CPU
进程数pids.max创建大量进程
IOio.max写爆磁盘或 IO 饥饿
清理cgroup.kill任务结束后残留子进程

6.6 no_new_privs 与 seccomp

no_new_privs 设置后会跨 fork/clone/execve 继承，阻止 exec 后获得新权限。seccomp filter 用 BPF 过滤系统调用，适合减少内核攻击面，例如限制挂载、进程调试、加载内核模块、性能探测等高风险 syscall。

seccomp 不是路径策略。它不能可靠表达“允许打开某个工作区路径，拒绝打开另一个宿主路径”。路径策略应该交给 mount namespace、LSM、Landlock、AppArmor/SELinux 或应用层 broker。

6.7 LSM：AppArmor、SELinux、Landlock

AppArmor 和 SELinux 是 Linux 上常见的 Mandatory Access Control。AppArmor 更偏路径 profile，SELinux 更偏 label/type enforcement。Landlock 值得重点学习，因为它是非特权进程也可以使用的访问控制机制，可以让 runner 自己把未来子进程限制在特定文件层级中。

6.8 从容器到强隔离

方案特点适合注意
bubblewrap轻量 namespace/mount 工具学习、本地 agent、无网命令执行策略要自己设计
Docker rootless镜像生态成熟，资源限制方便工程化 runner仍共享宿主 kernel；不要挂宿主控制接口

7七、Windows 沙箱机制详解

Windows 更像“给进程一个受限身份，并让所有 securable object 都按 token、SID、DACL、Integrity Level 做访问检查”。文件、注册表 key、进程、线程、named pipe、window station、desktop、Job Object 等大多是 securable object。

7.1 Restricted Token

Restricted Token 可以删除 privileges、把 SID 标为 deny-only、添加 restricting SID。访问对象时，Windows 会对 enabled SID 和 restricting SID 做双重检查。Agent runner 可以给每个 session 一个受限身份，并让 workspace ACL 只授权给这个身份。

普通进程：
  token = user + groups + privileges

受限进程：
  token = user + deny-only groups + removed privileges + sandbox restricting SID

workspace ACL：
  allow sandbox identity read/write

最终效果：
  只能访问明确授权给 sandbox identity 的资源

7.2 Mandatory Integrity Control

Windows 有 Low、Medium、High、System 等完整性级别。普通用户进程通常是 Medium，管理员提升后是 High，浏览器沙箱子进程常常是 Low。低完整性进程通常不能写中高完整性对象，也不能轻易影响更高完整性的进程。

7.3 AppContainer 与 LPAC

AppContainer 是 Windows 原生应用沙箱，基于 Package SID 和 Capability SID 做最小权限控制，覆盖 credential、device、file、registry、network、process、window 等隔离面。LPAC，即 Less Privileged AppContainer，比普通 AppContainer 更收紧，很多普通 AppContainer 默认可访问的资源也需要显式 capability。

7.4 Job Object

Job Object 可以把一组进程作为一个单元管理，限制内存、CPU、活跃进程数量、运行时间，并能终止 job 里的所有进程。它类似 Linux cgroup 的一部分功能，但不是文件系统或网络沙箱。

Windows agent worker:
  CreateJobObject
  SetInformationJobObject
    - memory limit
    - CPU rate
    - active process limit
    - kill-on-job-close
  CreateProcess suspended
  AssignProcessToJobObject
  ResumeThread
  timeout 后 TerminateJobObject

7.5 Windows Sandbox、Windows Container 与 Hyper-V isolation

Windows Sandbox 是轻量隔离桌面环境，适合运行未知 exe 和 GUI 测试。但它默认并不是 agent 最小权限模型；网络、剪贴板、映射目录都需要谨慎配置。Windows Container 有 process isolation 和 Hyper-V isolation。Process isolation 共享宿主 kernel，Hyper-V isolation 则给容器更强的 VM 边界。

8八、macOS 沙箱机制详解

macOS 更偏“签名应用声明少量能力”。你先给应用签名，开启 App Sandbox，然后通过 entitlement 恢复少量文件、网络、硬件、自动化能力。除此之外，还有 TCC、SIP、Hardened Runtime 和 XPC privilege separation。

8.1 App Sandbox 与 entitlements

启用 com.apple.security.app-sandbox 后，应用默认失去大多数系统交互能力，再通过 entitlement 恢复需要的能力。常见能力包括 user-selected file read/write、network.client/server、application-groups、inherit 等。

能力例子agent 含义
文件访问user-selected read-only / read-write用户选择项目目录后才允许访问
网络network.client / network.server只能粗粒度控制出站/入站，不是域名 allowlist
App Groupapplication-groups多个 helper 共享受控目录
继承inherit子进程继承父应用静态 sandbox 权限

8.2 TCC

TCC 保护用户隐私资源，例如 Desktop、Documents、Downloads、iCloud Drive、Photos、Contacts、Camera、Microphone、Screen Recording、Accessibility、Automation 和 Full Disk Access。GUI/computer-use agent 常常需要 Screen Recording、Accessibility、Automation，这些都必须由用户显式授权。

TCC 和 App Sandbox 是两套机制。App Sandbox 放行不代表 TCC 放行；TCC 放行也不代表 App Sandbox 放行。

8.3 SIP、Hardened Runtime 与 sandbox-exec

SIP 保护 Apple 提供的系统组件，限制 root 修改系统路径或 attach 受保护进程。Hardened Runtime 主要保护你的 app 自身运行时完整性，限制代码注入、JIT、动态库注入、调试等高风险能力。它们不是运行任意不可信代码的完整 sandbox。

macOS 还可以用 sandbox-exec / Seatbelt / SBPL 限制命令行进程，例如只允许读写 workspace。它适合本地 Mac agent 的轻量命令沙箱，但文档和调试体验不如 Linux namespace/container 直接。

9九、为什么第一站建议选 Linux

推荐先从 Linux 开始，并不是因为 Windows/macOS 不能做沙箱，而是因为 Linux 最接近现代 agent runner、容器、Kubernetes、gVisor、Kata、Firecracker 的底层模型。

10十、从零实现：推荐路线

不要一上来就从零写容器或 VM。更稳的路线是：先实现一个 agent sandbox runtime，而不是先实现一个隔离内核。

阶段 0：威胁模型
  明确 agent 能做什么，不能做什么

阶段 1：bubblewrap MVP
  workspace-only、no network、clearenv、timeout、stdout/stderr

阶段 2：Sandbox API
  exec/read/write/list/diff/export/destroy

阶段 3：资源和审计
  cgroup v2、audit log、artifact、diff、攻击测试

阶段 4：Docker runner
  镜像化、依赖管理、工程化执行

阶段 5：网络代理与凭据 broker
  allowlist、审计、短期 token、approval

阶段 6：gVisor / microVM
  多租户、不可信代码、强隔离
允许禁止
在 /workspace 读写文件默认联网
运行 shell、Python、Node、测试命令读取宿主用户数据
生成 artifacts写 workspace 外文件
返回 stdout/stderr/exit code/diff拿真实长期凭据
超时后清理进程树访问宿主 runtime 控制接口

11十一、Linux MVP：bubblewrap runner

bubblewrap 适合第一版学习实现。它可以显式构造 mount namespace、PID namespace、network namespace、tmpfs、只读 bind mount 等。

mkdir -p /tmp/agent-workspace
cat > /tmp/agent-workspace/main.py <<'PY'
import os
print("cwd:", os.getcwd())
print("home:", os.environ.get("HOME"))
print("hello from sandbox")
PY

bwrap \
  --die-with-parent \
  --new-session \
  --unshare-pid \
  --unshare-ipc \
  --unshare-uts \
  --unshare-cgroup \
  --unshare-net \
  --proc /proc \
  --dev /dev \
  --tmpfs /tmp \
  --dir /run \
  --ro-bind /usr /usr \
  --ro-bind-try /bin /bin \
  --ro-bind-try /sbin /sbin \
  --ro-bind-try /lib /lib \
  --ro-bind-try /lib64 /lib64 \
  --ro-bind-try /etc/ld.so.cache /etc/ld.so.cache \
  --bind /tmp/agent-workspace /workspace \
  --chdir /workspace \
  --clearenv \
  --setenv HOME /tmp \
  --setenv PATH /usr/local/bin:/usr/bin:/bin \
  --setenv LANG C.UTF-8 \
  python3 main.py

这个版本具备 /workspace 可写、/tmp 为沙箱 tmpfs、运行时目录只读、无网络、清空环境变量、隔离 PID/IPC/UTS/cgroup 视图等基本特性。

11.1 Python wrapper

@dataclass
class ExecResult:

class BubblewrapSandbox:
    def __init__(self, workspace: str | Path):
        self.workspace = Path(workspace).resolve()
        if not self.workspace.exists():
            raise FileNotFoundError(self.workspace)
        if not self.workspace.is_dir():
            raise NotADirectoryError(self.workspace)

    def _limit_child(self) -> None:
        resource.setrlimit(resource.RLIMIT_CPU, (10, 10))
        resource.setrlimit(resource.RLIMIT_AS, (2 * 1024**3, 2 * 1024**3))
        resource.setrlimit(resource.RLIMIT_FSIZE, (200 * 1024**2, 200 * 1024**2))
        resource.setrlimit(resource.RLIMIT_NPROC, (128, 128))

    def build_argv(self, argv: list[str], network: bool = False) -> list[str]:
        if not argv:
            raise ValueError("argv must not be empty")

        bwrap = [
            "bwrap", "--die-with-parent", "--new-session",
            "--unshare-pid", "--unshare-ipc", "--unshare-uts", "--unshare-cgroup",
            "--proc", "/proc", "--dev", "/dev", "--tmpfs", "/tmp", "--dir", "/run",
            "--ro-bind", "/usr", "/usr",
            "--ro-bind-try", "/bin", "/bin",
            "--ro-bind-try", "/sbin", "/sbin",
            "--ro-bind-try", "/lib", "/lib",
            "--ro-bind-try", "/lib64", "/lib64",
            "--ro-bind-try", "/etc/ld.so.cache", "/etc/ld.so.cache",
            "--bind", str(self.workspace), "/workspace",
            "--chdir", "/workspace",
            "--clearenv",
            "--setenv", "HOME", "/tmp",
            "--setenv", "PATH", "/usr/local/bin:/usr/bin:/bin",
            "--setenv", "LANG", "C.UTF-8",
            "--setenv", "PYTHONPYCACHEPREFIX", "/tmp/pycache",
        ]
        if not network:
            bwrap.append("--unshare-net")
        return bwrap + argv

    def exec(self, argv: list[str], timeout_s: int = 30, network: bool = False) -> ExecResult:
        started = time.monotonic()
        try:
            proc = subprocess.run(
                self.build_argv(argv, network=network),
                cwd=str(self.workspace),
                stdin=subprocess.DEVNULL,
                stdout=subprocess.PIPE,
                stderr=subprocess.PIPE,
                text=True,
                timeout=timeout_s,
                preexec_fn=self._limit_child,
            )
            return ExecResult(argv, proc.returncode, proc.stdout[-200_000:], proc.stderr[-200_000:], False, int((time.monotonic() - started) * 1000))
        except subprocess.TimeoutExpired as e:
            stdout = e.stdout or ""
            stderr = e.stderr or ""
            if isinstance(stdout, bytes):
                stdout = stdout.decode("utf-8", errors="replace")
            if isinstance(stderr, bytes):
                stderr = stderr.decode("utf-8", errors="replace")
            return ExecResult(argv, None, stdout[-200_000:], (stderr + "\n[timeout]")[-200_000:], True, int((time.monotonic() - started) * 1000))

12十二、工程化：Docker runner

bubblewrap 适合学习和本地轻量执行，但工程化时会遇到依赖打包问题。Docker/Podman 的优势是镜像生态成熟、语言运行时容易预装、资源限制参数现成、CI 集成简单。

docker run --rm \
  --network none \
  --user 1000:1000 \
  --read-only \
  --tmpfs /tmp:rw,nosuid,nodev,size=512m \
  --tmpfs /run:rw,nosuid,nodev,size=64m \
  --pids-limit 128 \
  --memory 2g \
  --cpus 2 \
  --cap-drop ALL \
  --security-opt no-new-privileges \
  -e HOME=/tmp \
  -e PYTHONPYCACHEPREFIX=/tmp/pycache \
  -v "$PWD/workspace:/workspace:rw" \
  -w /workspace \
  python:3.12-slim \
  python main.py

关键点是：--network none 默认无网，--read-only 让根文件系统只读，--cap-drop ALL 移除 capabilities，no-new-privileges 阻止 exec 后提权。不要使用 privileged 模式，不要使用 host network，不要挂宿主用户目录，不要把真实凭据放进环境变量。

13十三、Policy、审批、网络代理与凭据 broker

Agent sandbox 从“代码执行器”升级为“agent runtime”，关键在 policy、approval、egress proxy 和 credential broker。

13.1 命令策略

明显高风险的命令应该在执行前触发审批或拒绝，例如提权、挂载、修改网络、控制容器 runtime、发布包、推送代码、调用云 CLI 等。但命令黑名单只是辅助，真正的防线仍然是无网络、无长期凭据、无挂载权限、无高风险 capabilities、workspace-only。

13.2 网络代理

sandbox:
  没有全网访问
  只能连到本地代理或无网络

proxy:
  解析目标域名
  检查 allowlist
  阻断内网 CIDR 和 metadata endpoint
  记录请求日志
  注入短期凭据
  必要时触发用户审批

13.3 凭据 broker

不要把真实长期凭据放进沙箱。更好的模式是：

agent 生成 patch
  ↓
sandbox 返回 diff
  ↓
harness 展示 diff / 策略审核
  ↓
用户或自动规则批准
  ↓
broker 用真实凭据创建 PR 或执行外部动作
  ↓
sandbox 永远看不到长期凭据

14十四、攻击测试集与验收标准

在接入 agent 前，先写攻击测试。沙箱不是靠“看起来安全”，而是靠反复验证。

测试类别验证目标期望结果
文件读取逃逸尝试访问宿主用户目录、系统敏感区域、runtime 控制接口不可见或不可读
写越界尝试写入 workspace 外路径失败或只读
网络外传尝试连接公网、内网、metadata 地址默认失败
环境变量泄露查看沙箱环境变量只有最小必要变量
进程数滥用创建大量子进程被进程数限制拦住
内存滥用持续申请内存被内存限制拦住
CPU 滥用无限循环被 timeout 或 CPU quota 控制

验收清单：

  • exec(argv) 不使用裸 shell string
  • 默认 no network
  • 默认 clearenv
  • 默认 workspace-only
  • 没有宿主用户目录
  • 没有真实长期凭据
  • 有 timeout、内存、CPU、进程数限制
  • stdout/stderr 截断
  • 有 audit log
  • 有 file diff 或 hash snapshot
  • 高风险动作走 approval
