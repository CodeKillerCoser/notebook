所有权与借用心智模型

核心概念Option

1核心矛盾

Rust 不允许直接碰 Option<Box<T>> 里面的 Box<T>——碰了就意味着所有权转移，原变量就废了。

其他语言里直接 node.val 就行（背后是 GC 引用/指针拷贝），Rust 没有 GC，所有权的账必须算清楚。于是催生了 as_ref / as_mut / map / and_then / unwrap 这一整套工具方法。

2根本思路切换

其他语言思维：我有一个值 → 我直接用 Rust 思维： 我有没有所有权？ ├── 有 → 可以 unwrap / map / and_then（消费式） └── 没有 → 只能 as_ref / match / if let（借用式）
核心困境：想用"消费式"方法（unwrap）取出值，但又想保留原变量给后面用。这在 Rust 里是矛盾的。

解法只有两个

  • 路径 A：提前 clone，各自消费各自的
  • 路径 B：用借用式方法（as_ref、as_deref、match、if let），不消费原值

3两条路径对照

路径 A：消费式（变量用完即废）

let p: Option<Box<ListNode>> = ...;

// 一次性取出，之后 p 不能再用了
let val = p.unwrap().val;

// 一次性转换，生成新的 Option
let next: Option<Box<ListNode>> = p.and_then(|node| node.next);

路径 B：借用式（变量保留）

let p: Option<Box<ListNode>> = ...;

// 借出来看一眼，p 还在
let val = p.as_ref().map_or(0, |node| node.val);

// 用 match + 引用，同样不消费 p
let val = match &p {
    Some(node) => node.val,
    None => 0,
};

// if let 也是借用
if let Some(ref node) = p {
    println!("{}", node.val);
}

4动手前的第一问

我还要继续用这个变量吗？ ├── 不用了 → 消费式：unwrap / map / and_then / unwrap_or ... └── 还要用 → 借用式：as_ref / as_deref / match & / if let
记住：方法签名中 self（不是 &self）的参数一定会消费原值。看签名就能判断会不会 move。
