Send 和 Sync 是 Rust 线程安全体系的基石。它们是 marker trait — 没有方法要实现，标记了类型是否能在线程间安全传递。

1定义

// Send：价值所有权可以安全地转移到另一个线程
pub unsafe auto trait Send {}

// Sync：多个线程可以安全地通过共享引用访问
pub unsafe auto trait Sync {}
  • auto trait — 编译器自动为符合条件的类型实现，你通常不需要手动写
  • unsafe — 只有你能保证线程安全时才手动实现

2正向含义

Send  = "我可以把所有权安全地转移到另一个线程"
Sync  = "多个线程可以同时通过引用访问我"

3反向记忆法（更直观）

类型为什么不是 Send为什么不是 Sync
Rc<T>非原子引用计数，两个线程同时 clone() / drop() 会数据竞争同左，共享引用下计数仍不安全
RefCell<T>（RefCell 是 Send 当 T: Send）=借用计数非原子，两个线程同时 borrow_mut() 会崩
Mutex<T>当 T: Send当 T: Send（互斥保证单线程访问）
裸指针 *const T编译器不知道它指向谁、生命周期、所有权没有 &T 的保证
Arc<T>当 T: Send + Sync当 T: Send + Sync

4Send 和 Sync 的关系

如果 T: Sync，则 &T: Send。
如果 &T: Send，则 T: Sync。

逻辑：如果能多线程共享引用（Sync），那引用本身就能传到另一个线程（Send）。反之亦然。

5实际约束

use std::rc::Rc;
use std::thread;

// ❌ Rc 不是 Send，不能 move 进线程闭包
let a = Rc::new(42);
thread::spawn(move || {
    println!("{}", a);  // 编译错误：`Rc<i32>` cannot be sent between threads safely
});

// ❌ RefCell 不是 Sync，不能通过 Arc 共享
use std::sync::Arc;
use std::cell::RefCell;
let b = Arc::new(RefCell::new(42));
let b2 = b.clone();
thread::spawn(move || {
    b2.borrow_mut();  // 编译错误：`RefCell<i32>` cannot be shared between threads
});

6复合类型的发送规则

一个类型是 Send 当且仅当其所有字段都是 Send。Sync 同理。因此嵌套类型有"传染性"：

Arc<Mutex<Vec<Rc<String>>>>
                ↑
                包含 Rc → 整体不是 Send
                即使外面包了 Arc + Mutex 也不行

7C++ 对比

RustC++
Send / Sync — 编译期静态检查无等价物 — 依赖文档和约定
违反 → 编译错误违反 → 运行时数据竞争（UB）
Rc<T> 单线程 — 编译器阻止你跨线程shared_ptr<T> — 允许跨线程，但非原子计数是 UB
Arc<T> — 原子计数的 shared_ptrshared_ptr<T> 控制块本身有原子计数
关键区分：在 Rust 里，数据竞争不会出现，因为编译器在编译期就拒绝了所有跨线程不安全的组合。C++ 要靠你记住"shared_ptr 的控制块是线程安全的，但指向的 T 不是"。
