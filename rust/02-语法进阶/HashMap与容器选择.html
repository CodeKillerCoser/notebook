02 — HashMap 与容器选择

在滑动窗口等算法中，需要快速判断「某个字符是否已出现过」并记录位置。 Rust 提供了两种典型选择：HashMap（通用堆容器）和 [i32; 128]（专用栈数组）。它们的 API 设计和内存特性差异显著。

1. HashMap 核心 API 详解

1.1 get(&key) — 一次查找的完整生命周期

use std::collections::HashMap;

let mut map: HashMap<char, usize> = HashMap::new();

//            ┌── 为什么这里要 &ch？
//            │
if let Some(&prev) = map.get(&ch) {
//         ^^^^^         ^^^
//      (2) 剥引用     (1) 传引用
    start = start.max(prev + 1);
}
map.insert(ch, i);

这一行涉及两个 &，方向相反：

map.get(&ch) │ (1) 传入 &ch ───────┘ get 的签名：fn get<Q>(&self, k: &Q) -> Option<&V> 要求传 key 的引用，传 ch 值的话类型不匹配。 │ (2) 返回 Some(&prev) ◄──── 桶中找到值，返回指向桶内 usize 的指针 &prev 是模式匹配的解引用，剥掉 &，让 prev: usize

1.2 Some(prev) 也能编译？— 自动解引用

// 写法 A — 模式里剥引用
if let Some(&prev) = map.get(&ch) {
    start = start.max(prev + 1);   // prev: usize
}

// 写法 B — 引用保留，用时自动解
if let Some(prev) = map.get(&ch) {
    start = start.max(prev + 1);   // prev: &usize, 但 + 触发自动解引用
}

写法 B 能工作是因为 Rust 运算符解析时遇到 prev + 1： 


prev: &usize
  │
  │  prev + 1   →  找 &usize 的 Add 实现 → 没找到
  │                    ↓
  │               自动解引用 *prev: usize
  │                    ↓
  │               usize 有 Add<usize> ✅
  │                    ↓
  │               (*prev) + 1
  ▼
start.max((*prev) + 1)
✅ 建议用 Some(&prev)：在模式匹配阶段就把引用剥掉，让 prev 一诞生就是 usize 值。后续代码不需要心智负担去记"这是引用还是值"。 编译优化后两者机器码完全一样。

1.3 contains_key + [&k] — 两次查找

// ❌ 两次哈希计算
if map.contains_key(&c) {
    start = map[&c] + 1;   // 又一次定位桶
}

// ✅ 一次查找
if let Some(&prev) = map.get(&c) {
    start = start.max(prev + 1);
}
写法哈希计算次数Option 匹配
contains_key + [&k] 2 次隐式（panic 风险）
get(&k) 1 次显式（安全）

2. 栈数组 vs 堆容器：性能差异的根源

2.1 内存布局对比

HashMap<char, usize> [usize; 128] ───────────────────── ─────────────── 堆 (heap) 栈 (stack) ┌─────────┐ ┌─────┬─────┬─────┬─────┐ │ bucket │ ← 指针 │ 0 │ 0 │ 0 │ ... │ 共 128 个 usize │ bucket │ ← 可能跨 cache │ │ │ │ │ = 512 字节（64位） │ bucket │ line ├─────┴─────┴─────┴─────┤ │ ... │ │ 一整行在 L1 cache │ │ bucket │ ← rehash 时搬 └─────────────────────────┘ └─────────┘ 哈希+碰撞+扩容开销 基址+偏移 → 1 个 CPU 周期

2.2 操作开销拆解

操作HashMap[usize; 128]
查找 cnt[ch] 计算哈希 → 定位桶 → 比较 key *(base + ch * 8)
插入 get_or_insert 可能触发 rehash / 扩容 → 堆分配 栈上原地写入
Cache 行为 桶散列分布 → cache miss 概率高 紧凑 512 字节 → 大概率全在 L1

2.3 两种容器的正确场景


s: &str

   │
   ├── 仅含 ASCII 字符 (0~127)？
   │       │
   │       └── Yes → [i32; 128] 栈数组  【最快，O(n)】
   │                  ✅ 无堆分配、无哈希、cache 友好
   │                  ❌ 多字节字符逻辑错误
   │
   └── 需要通用 Unicode 支持？
           │
           └── Yes → HashMap<char, usize> 【正确，O(n)】
                      ✅ 正确识别所有 Unicode 字符
                      ⚠️ 有哈希开销，但现代硬件下完全可接受
结论：对 LeetCode 测试用例（纯 ASCII 字母），字节数组版利用了对输入的 先验知识 做了特化优化。作为通用算法，HashMap + char_indices 才是正确且足够快的选择。

3. 本次讨论的最终代码

use std::collections::HashMap;

struct Solution;

impl Solution {
    pub fn length_of_longest_substring(s: String) -> i32 {
        let mut last_seen: HashMap<char, usize> = HashMap::new();
        let mut start = 0;
        let mut max_len = 0;

        for (end, ch) in s.char_indices() {
            // 一次 get，同时完成「是否存在」和「取值」
            if let Some(&prev) = last_seen.get(&ch) {
                // 取 max 防止 start 回退（如 "abba"）
                start = start.max(prev + 1);
            }
            max_len = max_len.max(end - start + 1);
            last_seen.insert(ch, end);
        }

        max_len as i32
    }
}
