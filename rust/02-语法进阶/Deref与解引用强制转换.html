Rust 中所有 *ptr 解引用操作在底层都等价于 *(ptr.deref())。理解这个两层结构是理解智能指针的关键。

1Deref Trait 定义

pub trait Deref {
    type Target: ?Sized;
    fn deref(&self) -> &Self::Target;   // 返回引用，不是值
}

pub trait DerefMut: Deref {
    fn deref_mut(&mut self) -> &mut Self::Target;
}

2为什么 deref() 返回引用而不是值

三个原因：

  1. 不能转移所有权 — 如果 deref(self) -> Self::Target，智能指针被消耗后无法继续使用
  2. 保持借用语义 — *ptr 应该返回借用的值，否则第二次 *ptr 会拿到空数据
  3. 统一设计 — 编译器用 *(ptr.deref()) 统一了所有解引用，* 只负责"顺着引用取到值"
// 你写的：
let v = *box_ptr;

// 编译器展开的：
let v = *(box_ptr.deref());  // deref() 拿到 &T，* 拿到 T

3自动解引用规则：Deref Coercion

编译器只在特定场景自动插入解引用操作：

场景是否自动解引用能穿多层？
方法调用 .method()✅ 自动穿✅ 穿到找到方法为止
函数参数类型不匹配✅ Deref coercion✅ 递归穿
* 操作符✅ 一层❌ 只做一层
比较操作 == >❌ 不自动—

方法调用的自动解引用算法

编译器看到 receiver.method(args) 时，按以下顺序尝试：

1. 按值：     T           → 找方法
2. 自动引用： &T          → 找方法
3. 自动引用： &mut T      → 找方法
4. 解引用：   *T (= deref) → 回到步骤 1，用 Target 类型重复

走一遍完整流程：guard.push(4)

T = MutexGuard<Vec<i32>>

尝试 1: MutexGuard 有没有 .push()？       → 没有
尝试 2: &MutexGuard 有没有 .push()？       → 没有
尝试 3: &mut MutexGuard 有没有 .push()？   → 没有
尝试 4: Deref → *T = Vec<i32>，回到步骤 1

  T = Vec<i32>
  尝试 1: Vec 有没有 .push()？            → 没有 (push 要 &mut self)
  尝试 2: &Vec 有没有 .push()？            → 没有
  尝试 3: &mut Vec 有没有 .push()？        → 有！✅

穿越多层例子：box.len()

let b = Box::new(String::from("hello"));
let len = b.len();  // Box → String → str → .len()

编译器自动穿越两层 Deref：Box<String> → String → str，找到 str::len(&self)。

4关键陷阱

let b = Box::new(Box::new(42));
*b = 5;     // ❌ *b 是 Box<i32>，不能赋 i32
**b = 5;    // ✅ **b 是 i32

* 操作符只做一层解引用，不会自动穿越。

5演示：自定义智能指针

struct MyBox<T>(T);

impl<T> Deref for MyBox<T> {
    type Target = T;
    fn deref(&self) -> &T {
        &self.0
    }
}

let b = MyBox(42);
println!("{}", *b);     // *b → *(b.deref()) → 42
println!("{}", b.abs()); // 自动解引用 → i32::abs
核心记忆：编译器只会在 .（方法调用/字段访问）和函数传参时自动穿解引用链。* 操作符、比较、赋值解引用只帮你做一层。
