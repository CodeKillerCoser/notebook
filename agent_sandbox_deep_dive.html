一、总览：Agent 沙箱到底是什么

在 agent 系统里，sandbox 不是一句“请模型不要乱来”的提示词，也不只是一个 Docker 容器。更准确的理解是：把模型驱动的动作放进一个可丢弃、可限权、可审计、可审批的执行边界里。

Agent Sandbox = 隔离执行环境 + 权限策略 + 资源限制 + 网络出口治理 + 凭据代理 + 审计与恢复

这里有一个容易混淆的点：沙箱保护的不是模型内部的推理过程，而是模型发起的外部动作。模型可以“想”运行命令，但真正执行命令的必须是 runtime；模型可以“想”访问网站，但真正决定网络是否放行的必须是 egress proxy 或 policy engine；模型可以“想”使用 GitHub token，但真实 token 不应该直接放进沙箱。

OpenAI 的 Codex 文档也把 sandbox 和 approval 明确分开：sandbox 是技术边界，决定 agent 能读写哪里、命令能否联网；approval policy 决定越界前什么时候必须询问用户。[1] OpenAI 的 Sandbox Agents 文档则把 harness 和 sandbox 拆成控制面与执行面：harness 管 orchestration、tool routing、approval、state 和 trace；sandbox 管文件、命令、依赖、端口和可恢复状态。[2]

二、为什么 agent 比普通代码更需要沙箱

传统服务的执行路径通常是开发者提前写好、测试过、review 后部署。Agent 的不同点在于：它可能在运行时根据上下文生成命令、脚本、补丁、浏览器动作或 MCP 工具调用。也就是说，执行路径本身是动态的。

这会带来几类特殊风险：

风险例子主要防线
误操作模型误删文件、改错目录、覆盖配置workspace-only、只读根文件系统、diff review
Prompt injection网页、README、issue 诱导 agent 读取 secret 并外传默认禁网、文件隔离、输出审查、工具策略
供应链攻击pip install 或 npm install 运行恶意 postinstallsetup/agent 阶段分离、包源 allowlist、无长期凭据
主机逃逸恶意二进制攻击内核或容器 runtimeseccomp、LSM、gVisor、Kata、Firecracker、VM
凭据泄露读取 .env、~/.ssh、~/.aws、浏览器 cookieclearenv、禁止挂 HOME、credential broker
网络外传把项目源码或 token POST 到外部站点network namespace、egress proxy、domain allowlist
资源滥用fork bomb、无限循环、爆内存、写满磁盘cgroup、Job Object、timeout、quota
现实副作用发邮件、git push、部署、删云资源、下单tool broker、最小 scope、human-in-the-loop

不要把 prompt 当安全边界

“请不要读取敏感文件”是软约束。真正的安全边界应该是：沙箱里根本看不到敏感文件；即使模型被诱导，也没有网络外传通道；即使命令被允许，系统调用和资源仍然受限制。

Anthropic 在 Claude Code sandboxing 文章中也强调，文件系统隔离和网络隔离要同时存在。只有文件隔离但有网络，仍可能外传 workspace 里的敏感内容；只有网络隔离但没有文件隔离，agent 仍可能污染宿主环境或植入后门。[3]

三、成熟 agent sandbox 的运行时架构

一个成熟的 agent sandbox 不应该只是 subprocess.run() 外面套一层容器，而应该拆成四个平面：

层职责不要放什么
Control Plane模型调用、工具路由、任务状态、审批、trace、恢复不要把它和不可信代码放在同一个边界里
Execution Planeshell、Python、Node、浏览器、测试、构建、文件系统不要放长期 secret 和宿主 HOME
Policy Plane文件、网络、命令、凭据、资源、工具权限不要只靠正则黑名单
Artifact Planestdout/stderr、diff、报告、生成文件、日志、快照不要让沙箱任意写回真实系统
Agent Harness
  ├─ model loop
  ├─ tool schema
  ├─ approval workflow
  ├─ policy decision log
  └─ artifact manager
       ↓
Sandbox Manager
  ├─ create session
  ├─ mount workspace
  ├─ apply resource limits
  ├─ attach network policy
  ├─ run command
  └─ destroy / snapshot
       ↓
Sandbox Runtime
  ├─ /workspace       rw
  ├─ /tmp             rw tmpfs
  ├─ /usr /bin /lib   ro
  ├─ no host HOME
  ├─ no host docker.sock
  ├─ no raw secrets
  └─ no network by default

从 agent 工程角度看，关键接口不是“执行一段 shell 字符串”，而是一个明确的 session API：

SandboxSession:
  exec(argv, cwd, env, timeout, network_policy) -> ExecResult
  read_file(path) -> bytes
  write_file(path, bytes) -> void
  list_files(path) -> list
  diff() -> patch
  export_artifact(path) -> artifact
  destroy() -> void

这样做的好处是：agent harness 不需要知道底层是 bubblewrap、Docker、gVisor 还是 Firecracker。你可以先实现一个简单 runner，后续替换为更强隔离的后端。

四、要管的不是 shell，而是能力

很多人第一次做 agent sandbox 时，会把注意力集中在“禁止哪些命令”。这不够。因为模型可以用很多方式绕开字符串黑名单：用 Python 发 socket、用 Node 读文件、用 Git hook、用构建脚本、用 npm postinstall、用二进制工具。

更正确的模型是 capability-based security：显式建模 agent 拥有哪些能力。

filesystem:
  read:
    - /workspace
  write:
    - /workspace
    - /tmp
  deny:
    - /home
    - /root
    - /var/run/docker.sock

network:
  default: deny
  allow_domains: []
  block_private_cidrs: true
  block_metadata_service: true

process:
  user: nonroot
  timeout_seconds: 600
  memory: 4GiB
  cpu: 2
  pids: 256

secrets:
  raw_values_inside_sandbox: false
  injection: brokered_proxy
  ttl: short

approvals:
  require_for:
    - new_network_domain
    - write_outside_workspace
    - production_api_call
    - destructive_command

硬边界

  • namespace / container / VM
  • cgroup / Job Object
  • seccomp / LSM / AppContainer
  • network namespace / firewall / proxy

流程边界

  • approval
  • diff review
  • credential broker
  • audit log

MCP 工具也必须纳入这个能力模型。MCP 官方安全最佳实践强调 confused deputy、token passthrough、SSRF、本地 MCP server compromise、scope minimization 等风险；本地 MCP server 如果没有沙箱和用户确认，可能用 MCP client 的权限执行命令或外传数据。[4]

五、三平台实现哲学对比

Linux、Windows、macOS 都能做沙箱，但它们的底层哲学完全不同。

平台核心哲学关键机制最适合的 agent 形态
Linux给进程造一个新的小世界namespace、cgroup、capabilities、seccomp、AppArmor/SELinux/Landlock、container runtime、microVM云端 agent runner、代码执行器、CI-like coding agent、多租户 sandbox
Windows给进程一个受限身份，让它无法访问未授权对象Access Token、SID、DACL、Integrity Level、Restricted Token、AppContainer/LPAC、Job Object、Windows Container、Hyper-V、Windows SandboxWindows GUI agent、Win32 工具自动化、企业桌面自动化、Windows 容器任务
macOS签名应用声明少量 entitlement，其余默认被系统拒绝App Sandbox、entitlements、security-scoped bookmarks、TCC、SIP、Hardened Runtime、Seatbelt/sandbox-exec、XPC本地 Mac agent、IDE agent、保护用户文件和隐私资源

Linux 是“看不到”；Windows 是“看到了也打不开”；macOS 是“没 entitlement 和用户授权就不给”。

六、Linux 沙箱机制详解

Linux 沙箱不是一个单独 API，而是一组内核原语的组合。Linux namespace 把全局资源包装成隔离实例，namespace 内的进程看到的是自己的进程树、网络栈、mount 表、IPC、hostname、用户 ID 等。Linux manual page 明确列出 cgroup、IPC、network、mount、PID、time、user、UTS 等 namespace 类型。[5]

6.1 Mount namespace：文件系统视图

Mount namespace 是 agent sandbox 最核心的一层。它决定沙箱里能看到哪些目录、哪些目录只读、哪些目录可写。

sandbox root
├── workspace/      rw bind mount，agent 的工作区
├── tmp/            tmpfs，任务结束销毁
├── usr/            ro bind mount，运行时程序
├── bin/            ro bind mount
├── lib/            ro bind mount
├── proc/           沙箱自己的 proc
└── dev/            最小设备节点

不要挂载：

  • $HOME、~/.ssh、~/.aws、~/.config
  • 浏览器 profile、cookie、Keyring、云 CLI 配置
  • /var/run/docker.sock
  • 宿主 /tmp、/var、/etc 的可写视图

Mount namespace 比应用层路径检查更强。应用层 path check 可能被 symlink、..、hardlink、race condition 绕过；mount namespace 则让敏感路径在内核解析阶段就不存在。

6.2 PID namespace：进程树隔离

PID namespace 让沙箱里的进程看到自己的 PID 空间。这样 ps aux 看不到宿主机进程，kill 不能直接杀宿主机进程，任务结束时也更容易清理整棵子进程树。

但 PID namespace 不限制 fork bomb，所以必须配合 pids.max 或 RLIMIT_NPROC。

6.3 Network namespace：网络栈隔离

Network namespace 隔离网卡、路由表、防火墙规则、端口空间等。Agent 的默认策略应该是无网络，因为网络是数据外传的主要通道。

默认：
  no network

需要安装依赖时：
  setup phase 允许有限包源
  agent phase 重新回到 no network

需要访问外部 API 时：
  sandbox -> egress proxy -> allowlist / audit / token broker -> internet

6.4 User namespace 与 capabilities

User namespace 允许沙箱内的 root 映射成宿主机上的普通用户。这是 rootless container 的基础：沙箱内看起来是 UID 0，但宿主机上不是 root。

Linux capabilities 把传统 root 权限拆成更细的能力，例如网络管理、mount、ptrace、加载内核模块等。Linux manual page 说明 capabilities 是 per-thread 属性，用于把超级用户权限分割为独立单元。[6] Agent sandbox 默认应该 drop 所有 capabilities，尤其不要给 CAP_SYS_ADMIN、CAP_NET_ADMIN、CAP_SYS_PTRACE、CAP_DAC_OVERRIDE、CAP_SYS_MODULE。

6.5 cgroup v2：资源限制

Namespace 解决“看到什么”，cgroup 解决“能用多少”。Linux kernel 文档把 cgroup v2 描述为用户态可见的资源控制接口，用来分层组织进程，并通过 controller 分配或限制资源。[7]

资源cgroup v2 控制项agent 风险
内存memory.max、memory.swap.max爆内存拖垮宿主
CPUcpu.max无限循环或构建任务吃满 CPU
进程数pids.maxfork bomb
IOio.max写爆磁盘或 IO 饥饿
清理cgroup.kill任务结束后残留子进程

6.6 no_new_privs 与 seccomp

no_new_privs 是一个安全位，设置后跨 fork/clone/execve 继承且不可取消，能防止 exec 后获得新权限。[8]

seccomp filter 用 BPF 程序过滤系统调用。Linux kernel 文档说明过滤器输入包括 syscall number 和参数。[9] 它适合减少内核攻击面，例如禁止 mount、ptrace、kexec_load、init_module、bpf、perf_event_open 等危险 syscall。

seccomp 不是路径策略

seccomp 不能可靠表达“允许打开 /workspace/a，拒绝打开 ~/.ssh/id_rsa”。路径策略应该交给 mount namespace、LSM、Landlock、AppArmor/SELinux 或应用层 broker。

6.7 LSM：AppArmor、SELinux、Landlock

AppArmor 和 SELinux 是 Linux 上常见的 Mandatory Access Control。AppArmor 更偏路径 profile，SELinux 更偏 label/type enforcement。Landlock 则非常适合学习和自研 runner，因为它是非特权进程也可以使用的 access control 机制。Linux kernel 文档说 Landlock 目标是限制一组进程的 ambient rights，并可在已有访问控制之外叠加规则。[10]

6.8 从容器到强隔离

方案特点适合注意
bubblewrap轻量 namespace/mount 工具学习、本地 agent、无网命令执行策略要自己设计
Docker rootless镜像生态成熟，资源限制方便工程化 runner仍共享宿主 kernel；别挂 docker.sock
gVisor用户态内核拦截 syscall多租户代码执行，想保留容器体验兼容性和性能要测试
Kata / FirecrackermicroVM / VM 边界高风险、多租户、不可信代码镜像、网络、快照和调度更复杂

Docker rootless mode 会让 Docker daemon 和容器以非 root 用户运行，Docker 文档说明 rootless mode 在 user namespace 中执行 daemon 和 container。[11] Docker 默认 seccomp profile 是 allowlist 模型，用于阻止不在 allowlist 中的 syscall。[12] gVisor 提供 runsc 作为 OCI runtime，目标是在应用与宿主内核之间提供 Linux-compatible sandbox。[13] Firecracker 则是 lightweight virtual machine / microVM 路线。[14]

七、Windows 沙箱机制详解

Windows 的安全模型和 Linux 完全不同。Linux 更像“给进程造一个新的世界”；Windows 更像“给进程一个受限身份，并让所有 securable object 都按 token、SID、DACL、Integrity Level 做访问检查”。

7.1 Access Token、SID、DACL

Windows 进程带有 access token，里面包含用户 SID、组 SID、privileges、integrity level、AppContainer SID、capability SID 等。文件、注册表 key、进程、线程、named pipe、window station、desktop、Job Object 等大多是 securable object。访问时，系统会把 token 和对象的 security descriptor / DACL / mandatory label 做匹配。

7.2 Restricted Token

Restricted Token 可以删除 token 里的 privileges、把 SID 标为 deny-only、添加 restricting SID。Microsoft 文档说明，系统会对 enabled SID 和 restricting SID 做两次访问检查，只有两次都允许才授予访问。[15]

普通进程：
  token = Alice + Users + Developers + privileges

受限进程：
  token = Alice + deny-only groups + removed privileges + Sandbox_123 restricting SID

workspace ACL：
  allow Sandbox_123 read/write

最终效果：
  只能访问明确授权给 Sandbox_123 的资源

这适合做轻量本地 Windows command runner，但它不会自动给你独立文件系统、独立网络栈或容器视图。

7.3 Mandatory Integrity Control

Windows 有 Low、Medium、High、System 等完整性级别。普通用户进程通常是 Medium，管理员提升后是 High，受限浏览器/沙箱子进程常常是 Low。Low integrity 进程通常不能写 Medium integrity 对象，也不能轻易影响更高完整性的进程。

对 agent runner，可以把 worker 降到 Low Integrity，再给 workspace 设置对应 ACL。但这仍然不是完整容器。

7.4 AppContainer 与 LPAC

AppContainer 是 Windows 原生应用沙箱。Microsoft 文档说 AppContainer isolation 的目标是隔离应用，基于最小权限控制资源访问，并覆盖 credential、device、file、network、process、window 等隔离面。[16]

隔离面意义
Credential isolation应用不能直接冒充用户拿凭据
Device isolation摄像头、麦克风、定位等默认阻止，按 capability 授权
File / Registry isolation只能访问被授权的文件和注册表区域
Network isolationInternet、Intranet、server 能力按 capability 授权
Process / Window isolation限制影响其他进程和窗口

LPAC，即 Less Privileged AppContainer，比普通 AppContainer 更收紧，很多普通 AppContainer 默认可访问的资源也需要显式 capability。做 Windows 本地 agent 时，AppContainer/LPAC 适合运行不可信解析器、插件、浏览器子进程或命令 worker。

7.5 Job Object：Windows 的资源和进程树控制

Job Object 可以把一组进程作为一个单元管理，限制 working set、优先级、运行时间，并能终止 job 里的所有进程。Microsoft 文档说明，job object 用来控制关联进程的属性，对 job object 的操作会影响所有关联进程。[17]

Windows agent worker:
  CreateJobObject
  SetInformationJobObject
    - memory limit
    - CPU rate
    - active process limit
    - kill-on-job-close
  CreateProcess suspended
  AssignProcessToJobObject
  ResumeThread
  timeout 后 TerminateJobObject

Job Object 类似 Linux cgroup 的一部分功能，但它不是文件系统或网络沙箱。

7.6 Windows Sandbox、Windows Container 与 Hyper-V isolation

Windows Sandbox 是轻量隔离桌面环境，适合运行未知 exe 和 GUI 测试。Microsoft 文档说明 Windows Sandbox 提供轻量桌面环境，用于安全运行应用。[18] 但它默认并不是 agent 最小权限模型；网络、剪贴板、映射目录都需要谨慎配置。

<Configuration>
  <Networking>Disable</Networking>
  <ClipboardRedirection>Disable</ClipboardRedirection>
  <MappedFolders>
    <MappedFolder>
      <HostFolder>C:\AgentWorkspace</HostFolder>
      <SandboxFolder>C:\Workspace</SandboxFolder>
      <ReadOnly>false</ReadOnly>
    </MappedFolder>
  </MappedFolders>
  <MemoryInMB>4096</MemoryInMB>
</Configuration>

Windows Container 有 process isolation 和 Hyper-V isolation。Process isolation 更像共享内核的容器，Hyper-V isolation 则给容器一个更强的 VM 边界。对真正不可信的 Windows workload，Hyper-V isolation 或 Windows Sandbox/VM 更合适。

7.7 Windows 上的推荐 agent sandbox 形态

形态组合适合

八、macOS 沙箱机制详解

macOS 的哲学更偏“签名应用声明少量能力”。你先给应用签名，开启 App Sandbox，然后通过 entitlement 恢复少量文件、网络、硬件、自动化能力。除此之外，还有 TCC、SIP、Hardened Runtime 和 XPC privilege separation。

8.1 App Sandbox 与 entitlements

Apple 的 App Sandbox 是基于 entitlement 的能力模型。启用 com.apple.security.app-sandbox 后，应用默认失去大多数系统交互能力，再通过 entitlement 恢复需要的能力。Apple 的 App Sandbox 文档需要浏览器 JavaScript，但官方页面明确对应这一安全能力。[19]

常见 entitlement 包括：

能力例子agent 含义
文件访问user-selected read-only / read-write用户选择项目目录后才允许访问
网络network.client / network.server只能粗粒度控制出站/入站，不是域名 allowlist
App Groupapplication-groups多个 helper 共享受控目录
继承com.apple.security.inherit子进程继承父应用静态 sandbox 权限

8.2 TCC：透明、同意、控制

TCC 保护用户隐私资源。Apple Platform Security 文档说明，macOS 10.15 或更高版本要求 app 在访问 Documents、Downloads、Desktop、iCloud Drive、network volumes 之前获得用户同意；Full Disk Access、Accessibility、Automation 等也需要用户在隐私设置中显式授权。[20]

这对本地 GUI agent 特别重要，因为它可能需要：

  • Screen Recording：看屏幕
  • Accessibility：控制鼠标键盘和 UI 元素
  • Automation / Apple Events：控制其他 app
  • Full Disk Access：访问更广泛文件系统

TCC 和 App Sandbox 是两套机制

App Sandbox 放行不代表 TCC 放行；TCC 放行也不代表 App Sandbox 放行。做 Mac agent 时，两者要同时设计。

8.3 SIP 与 Hardened Runtime

SIP，即 System Integrity Protection，保护 Apple 提供的系统组件，限制即使是 root 用户也不能随便修改系统路径或 attach 受保护进程。Apple 文档说明 SIP 会保护系统位置并限制运行时 attach。[21]

Hardened Runtime 主要保护你的 app 自身运行时完整性，限制代码注入、JIT、动态库注入、调试等高风险能力。Apple 文档把它描述为与 SIP 一起保护软件运行时完整性的机制。[22]

但要注意：SIP 和 Hardened Runtime 不是“运行任意不可信代码”的完整 sandbox。它们是系统保护和应用自保护层。

8.4 sandbox-exec / Seatbelt / SBPL

macOS 还有命令行级的 sandbox-exec，可以用 profile 限制某个命令。其 profile 语言通常称为 SBPL。它适合本地 Mac agent 限制 shell 命令只能读写 workspace，但文档和调试体验不如 Linux namespace/container 直接。

(allow file-read*

8.5 macOS 上的推荐 agent 架构

Main App
  - UI
  - 用户授权
  - approval
  - TCC prompts
  - security-scoped bookmarks
  ↓
XPC Service / Helper

如果目标是运行大量不可信代码、包管理器、浏览器和测试，macOS 原生沙箱通常不是最佳执行层。更常见的产品化方案是：Mac app 作为控制面，真正的 code execution 放到 Linux VM、容器或远程 sandbox。

九、为什么第一站建议选 Linux

推荐先从 Linux 开始，并不是因为 Windows/macOS 不能做沙箱，而是因为你的目标有两部分：给自己的 agent 项目实现一个可用执行环境，同时深入掌握沙箱细节。

Linux 的优势在于它最接近现代 agent runner、容器、Kubernetes、gVisor、Kata、Firecracker 的底层模型。

学习目标Linux 对应机制后续工程路线

资源限制cgroup v2container resource limits、K8s requests/limits

权限拆分capabilities、user namespacerootless Docker、Podman
访问控制AppArmor、SELinux、Landlock企业安全 profile、runtime policy

Windows 更适合后续做 Windows GUI/Win32 agent，macOS 更适合后续做本地 Mac agent、IDE agent 和用户隐私资源保护。Linux 则是通用云端 agent execution runtime 的最短路径。

十、从零实现：推荐路线

不要一上来就从零写容器或 VM。更稳的路线是：先实现一个 agent sandbox runtime，而不是先实现一个隔离内核。

十一、Linux MVP：bubblewrap runner

bubblewrap 非常适合你的第一版学习实现。它是一个低层 unprivileged sandboxing 工具，可以显式构造 mount namespace、PID namespace、network namespace、tmpfs、只读 bind mount 等。[23]

11.1 最小命令

PY

  --unshare-pid \
  --unshare-ipc \
  --unshare-uts \
  --unshare-cgroup \
  --unshare-net \

11.2 Python wrapper

from __future__ import annotations

import json
import resource
import subprocess
import tempfile
import time
from dataclasses import dataclass
from pathlib import Path


@dataclass
class ExecResult:
    argv: list[str]
    exit_code: int | None
    stdout: str
    stderr: str
    timed_out: bool
    duration_ms: int


class BubblewrapSandbox:
    def __init__(self, workspace: str | Path):
        self.workspace = Path(workspace).resolve()
        if not self.workspace.exists():
            raise FileNotFoundError(self.workspace)
        if not self.workspace.is_dir():
            raise NotADirectoryError(self.workspace)

    def _limit_child(self) -> None:
        # 学习版 rlimit。生产版建议换成 cgroup v2。
        resource.setrlimit(resource.RLIMIT_CPU, (10, 10))
        resource.setrlimit(resource.RLIMIT_AS, (2 * 1024**3, 2 * 1024**3))
        resource.setrlimit(resource.RLIMIT_FSIZE, (200 * 1024**2, 200 * 1024**2))
        resource.setrlimit(resource.RLIMIT_NPROC, (128, 128))

    def build_argv(self, argv: list[str], network: bool = False) -> list[str]:
        if not argv:
            raise ValueError("argv must not be empty")

        bwrap = [
            "bwrap",
            "--die-with-parent",
            "--new-session",
            "--unshare-pid",
            "--unshare-ipc",
            "--unshare-uts",
            "--unshare-cgroup",
            "--proc", "/proc",
            "--dev", "/dev",
            "--tmpfs", "/tmp",
            "--dir", "/run",
            "--ro-bind", "/usr", "/usr",
            "--ro-bind-try", "/bin", "/bin",
            "--ro-bind-try", "/sbin", "/sbin",
            "--ro-bind-try", "/lib", "/lib",
            "--ro-bind-try", "/lib64", "/lib64",
            "--ro-bind-try", "/etc/ld.so.cache", "/etc/ld.so.cache",
            "--bind", str(self.workspace), "/workspace",
            "--chdir", "/workspace",
            "--clearenv",
            "--setenv", "HOME", "/tmp",
            "--setenv", "PATH", "/usr/local/bin:/usr/bin:/bin",
            "--setenv", "LANG", "C.UTF-8",
            "--setenv", "PYTHONPYCACHEPREFIX", "/tmp/pycache",
        ]

        if not network:
            bwrap.append("--unshare-net")

        return bwrap + argv

    def exec(self, argv: list[str], timeout_s: int = 30, network: bool = False) -> ExecResult:
        full_argv = self.build_argv(argv, network=network)
        started = time.monotonic()

        try:
            proc = subprocess.run(
                full_argv,
                cwd=str(self.workspace),
                stdin=subprocess.DEVNULL,
                stdout=subprocess.PIPE,
                stderr=subprocess.PIPE,
                text=True,
                timeout=timeout_s,
                preexec_fn=self._limit_child,
            )
            return ExecResult(
                argv=argv,
                exit_code=proc.returncode,
                stdout=proc.stdout[-200_000:],
                stderr=proc.stderr[-200_000:],
                timed_out=False,
                duration_ms=int((time.monotonic() - started) * 1000),
            )
        except subprocess.TimeoutExpired as e:
            stdout = e.stdout or ""
            stderr = e.stderr or ""
            if isinstance(stdout, bytes):
                stdout = stdout.decode("utf-8", errors="replace")
            if isinstance(stderr, bytes):
                stderr = stderr.decode("utf-8", errors="replace")
            return ExecResult(
                argv=argv,
                exit_code=None,
                stdout=stdout[-200_000:],
                stderr=(stderr + "\n[timeout]")[-200_000:],
                timed_out=True,
                duration_ms=int((time.monotonic() - started) * 1000),
            )


if __name__ == "__main__":
    ws = Path(tempfile.mkdtemp(prefix="agent-sbx-"))
    (ws / "main.py").write_text("print('hello from sandbox')\n")

    sbx = BubblewrapSandbox(ws)
    result = sbx.exec(["python3", "main.py"])
    print(json.dumps(result.__dict__, indent=2, ensure_ascii=False))

这个 wrapper 的关键原则：

  • 用 argv list，不拼 shell 字符串
  • 默认无网络
  • 清空环境变量
  • 只挂 workspace
  • stdout/stderr 截断
  • 有 timeout
  • 有基础资源限制
  • stdin 关闭

11.3 文件 API 的 path safety

即使 OS 沙箱存在，你暴露给模型的 read_file、write_file 仍然要做路径检查。

from pathlib import Path


def resolve_inside_workspace(workspace: Path, user_path: str) -> Path:
    root = workspace.resolve()
    target = (root / user_path).resolve()

    if target != root and root not in target.parents:
        raise PermissionError(f"path escapes workspace: {user_path}")

    return target


def read_file(workspace: Path, user_path: str) -> bytes:
    target = resolve_inside_workspace(workspace, user_path)
    return target.read_bytes()

但要记住：这个函数只能保护你自己的文件工具；保护不了 shell 里直接调用 open() 的程序。shell 的文件访问必须靠 mount namespace、container、Landlock、AppArmor/SELinux 或 microVM。

十二、工程化：Docker runner

bubblewrap 适合学习和本地轻量执行，但工程化时你很快会遇到依赖打包问题。Docker/Podman 的优势是镜像生态成熟、语言运行时容易预装、资源限制参数现成、CI 集成简单。

docker run --rm \
  --network none \
  --user 1000:1000 \
  --read-only \
  --tmpfs /tmp:rw,nosuid,nodev,size=512m \
  --tmpfs /run:rw,nosuid,nodev,size=64m \
  --pids-limit 128 \
  --memory 2g \
  --cpus 2 \
  --cap-drop ALL \
  --security-opt no-new-privileges \
  -e HOME=/tmp \
  -e PYTHONPYCACHEPREFIX=/tmp/pycache \
  -v "$PWD/workspace:/workspace:rw" \
  -w /workspace \
  python:3.12-slim \
  python main.py
参数作用
--network none默认无网
--user 1000:1000不以 root 运行
--read-only根文件系统只读
--tmpfs /tmp给临时目录可写空间
--pids-limit限制 fork bomb
--memory / --cpus限制资源
--cap-drop ALL移除 Linux capabilities
no-new-privileges阻止 exec 后提权

Docker 不是自动安全

不要使用 --privileged，不要使用 host network，不要挂宿主 HOME，不要挂 /var/run/docker.sock，不要把真实 secret 放进环境变量，不要关闭默认 seccomp。

十三、Policy、审批、网络代理与凭据 broker

一个 agent sandbox 真正从“代码执行器”升级为“agent runtime”，关键在 policy、approval、egress proxy 和 credential broker。

13.1 命令策略

明显高风险命令应该在执行前触发审批或拒绝：

sudo, su, mount, umount, unshare, nsenter, chroot, pivot_root
modprobe, insmod, rmmod, iptables, nft, ip link
ssh, scp, rsync remote, curl, wget, nc, socat
docker, podman, kubectl, helm, aws, gcloud, az
git push, npm publish, twine upload
rm -rf /, chmod +s, chown root

但命令黑名单只是辅助。真正的防线仍然是：无网络、无 secret、无挂载权限、无 capabilities、workspace-only。

13.2 网络代理

sandbox:
  没有全网访问
  只能连到本地代理或无网络

proxy:
  解析目标域名
  检查 allowlist
  阻断内网 CIDR 和 metadata endpoint
  记录请求日志
  注入短期凭据
  必要时触发用户审批

不要把网络 allowlist 写进 prompt。它应该是机器可执行的 policy。

13.3 凭据 broker

不要这样做：

docker run -e GITHUB_TOKEN=ghp_xxx ...
docker run -e AWS_SECRET_ACCESS_KEY=xxx ...

应该这样做：

agent 生成 patch
  ↓
sandbox 返回 diff
  ↓
harness 展示 diff / 策略审核
  ↓
用户或自动规则批准
  ↓
broker 用真实 GitHub token 创建 PR
  ↓
sandbox 永远看不到 token

这个设计原则可以泛化到 GitHub、云 API、数据库、邮件、Slack、钉钉、部署系统：真实凭据留在 trusted harness / broker，sandbox 只拿短期、受限、可撤销的能力。

13.4 审计日志

每次执行至少记录：

  "session_id": "sbx_001",

十四、攻击测试集与验收标准

在接入 agent 前，先写攻击测试。沙箱不是靠“看起来安全”，而是靠反复验证。

14.1 文件读取逃逸

ls /home
cat /etc/shadow
cat ~/.ssh/id_rsa
cat ~/.aws/credentials
ls /var/run/docker.sock

期望：宿主 HOME 不存在或不可读；SSH/AWS 凭据不可见；docker.sock 不存在。

14.2 写越界

echo pwned > /etc/cron.d/pwn
echo pwned > /usr/bin/x
echo pwned > /home/user/.bashrc

期望：Permission denied、Read-only filesystem 或 No such file or directory。

14.3 网络外传

curl https://example.com
python3 - <<'PY'
import socket
socket.create_connection(("1.1.1.1", 53), timeout=3)
PY

期望：默认失败。只有在 policy 显式允许的阶段和域名才可以成功。

14.4 环境变量泄露

env

期望：没有 OPENAI_API_KEY、GITHUB_TOKEN、AWS_SECRET_ACCESS_KEY、DATABASE_URL、NPM_TOKEN。

14.5 fork bomb、内存炸弹、CPU 无限循环

python3 - <<'PY'
import os
while True:
    os.fork()
PY

python3 - <<'PY'
x = []
while True:
    x.append("x" * 100_000_000)
PY

python3 - <<'PY'
while True:
    pass
PY

期望：进程数、内存、CPU 或 timeout 生效；宿主机正常；runner 能清理残留进程。

14.6 验收清单

  • exec(argv) 不使用裸 shell string
  • 默认 no network
  • 默认 clearenv
  • 默认 workspace-only
  • 没有宿主 HOME
  • 没有 docker.sock
  • 没有真实 secret
  • 有 timeout、内存、CPU、进程数限制
  • stdout/stderr 截断
  • 有 audit log
  • 有 file diff 或 hash snapshot
  • 高风险动作走 approval
